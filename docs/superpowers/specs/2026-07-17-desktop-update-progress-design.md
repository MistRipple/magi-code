# 桌面更新下载进度设计

## 目标

在桌面端下载更新时，让用户明确知道更新正在进行、已完成多少以及是否仍在等待网络数据。设置面板和启动时更新提示使用同一套进度语义与视觉反馈，避免两个入口表现不一致。

## 当前问题

更新底层已经从 Tauri updater 的 `Started` 和 `Progress` 事件中累计下载字节，并计算百分比；但两个界面只显示文本状态，没有可视化进度轨道。服务端没有提供 `contentLength` 时，当前状态可以保持无百分比，但缺少明确的不确定进度反馈。

## 方案评估

### 方案 A：只在设置按钮中显示百分比

改动最小，但用户离开设置面板后无法确认更新状态，也无法表达未知总大小，不采用。

### 方案 B：设置面板和启动提示共用统一进度组件（推荐）

保留现有 `DesktopUpdateProgress` 数据契约，在两个入口中统一展示进度条、百分比和字节信息。已知总大小时显示确定进度；未知总大小时显示不确定动画。下载完成后沿用现有重启流程，失败时保留错误和重试行为。该方案只扩展展示层，不重复实现下载状态机。

### 方案 C：把下载进度提升为全局独立 store

可以让更多页面订阅进度，但当前只有两个更新入口，新增全局生命周期和状态同步复杂度没有实际收益，暂不采用。

## 设计

### 数据与状态

- `DesktopUpdateProgress` 继续作为唯一下载进度模型。
- `downloadedBytes` 始终使用非负整数。
- `contentLength` 仅在有效正数时存在。
- `percent` 仅在总大小已知时存在，并限制在 `0..100`。
- `percent` 缺失时，UI 使用不确定进度动画，不显示伪造的数值。
- `Started` 事件重置累计下载量；`Progress` 事件累加分块大小；结束事件只负责把进度收口到已知完成值，不改变安装和重启流程。

### 设置面板

- 更新按钮保持现有状态文字和交互行为。
- 安装中状态在按钮下方显示紧凑进度条，避免改变顶部布局宽度。
- 总大小已知时显示百分比；总大小未知时显示“正在下载更新”及不确定动画。
- 安装中禁用重复检查和重复安装操作。

### 启动更新提示

- 更新提示卡片在安装中状态显示同一套进度条。
- 保留更新版本、下载状态和错误重试入口。
- 进度条宽度跟随卡片内容区域，手机视口下不溢出、不遮挡主要内容。
- 两处组件只共享数据语义和样式规则，不复制下载实现。

### 错误与生命周期

- 检查更新失败仍保持启动提示静默，设置面板显示可重试状态。
- 下载或安装失败显示错误状态，保留重试按钮。
- 组件卸载时继续关闭更新句柄，避免资源泄漏。
- 不引入第二套下载流程、回退逻辑或模拟进度。

## 测试与验收

自动化验证：

- 扩展 `desktop-updater-golden.mjs`，验证确定进度、不确定进度和百分比上限。
- 验证设置面板和启动提示都包含安装中进度条及不确定进度样式。
- 运行 `npm --prefix web run check`、更新相关 Golden 测试和完整前端测试。

真实浏览器验收：

- 在桌面入口打开设置，确认发现更新、开始下载、确定进度和失败重试状态。
- 确认启动更新提示在离开设置后仍能展示进度。
- 在窄屏视口检查进度条和按钮不溢出。
- 使用真实发布清单验证下载完成后仍按现有流程重启，不改变更新结果。

## 主线模型列表刷新补充

主线模型选择器的模型列表由当前主模型连接配置即时拉取，但打开选择器后会复用本次已加载结果。新增列表标题右侧的无文字刷新图标：点击后强制重新请求 `/api/settings/models/fetch`，不改变当前会话模型或推理强度。

- 刷新期间按钮进入等待态，禁止重复请求，并保留已显示列表，避免界面闪空。
- 请求成功后替换列表；如果返回空列表，显示既有空状态。
- 请求失败时保留旧列表并显示错误提示及原有重试入口。
- 刷新按钮使用 `refresh` 图标、明确的中英文无障碍标题，不增加新的 API 或缓存层。
- 真实浏览器验收需确认点击一次只发起一次请求、列表更新可见、当前选中项不被清除。
