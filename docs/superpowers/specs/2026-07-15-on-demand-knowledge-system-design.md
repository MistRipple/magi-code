# Magi 按需知识系统设计

## 目标

将 ADR、FAQ、Learning 从“可保存但很少被使用的数据”收敛为按需调用的项目能力：没有知识需求时不检索、不注入知识内容；存在架构决策、历史约束、已知问题或可复用经验需求时，能够在当前 workspace 内准确检索、受控注入并留下可观测记录。

## 非目标

- 不引入向量数据库、Embedding 服务或新的外部模型依赖。
- 不在每轮普通对话中固定查询或固定注入知识。
- 不让历史知识覆盖当前用户输入、当前任务目标或安全规则。
- 不把代码索引与 ADR、FAQ、Learning 混成同一套展示和注入策略。

## 核心原则

1. 知识库是能力，不是常驻上下文。
2. 没有触发知识需求时，知识内容 token 开销为零。
3. 普通主线与任务执行共享同一套意图判定、检索、排序和内容投影规则。
4. 所有查询必须绑定已注册 workspace；不得回退到默认 workspace，也不得跨 workspace 返回结果。
5. ADR、FAQ、Learning 使用不同的匹配权重和内容投影方式。
6. 查询、命中、注入和消费是不同状态，诊断必须分别记录。

## 架构

在 `magi-context-runtime` 增加统一的按需知识选择边界，负责以下职责：

- `KnowledgeIntentDetector`：判断当前输入是否需要项目知识。
- `KnowledgeContextSelector`：构造 workspace 级查询、调用 `KnowledgeStore`、应用阈值和预算。
- `KnowledgeContextRenderer`：按知识类型生成模型可读内容。
- `KnowledgeUsageTrace`：输出查询、命中和注入摘要，供事件总线和前端诊断使用。

普通主线和任务执行都只能通过该边界获得自动知识上下文。`knowledge_query` 工具继续保留，作为模型主动扩大查询范围、指定类型或获取更多内容的入口，并复用相同的检索和投影规则。

## 按需触发

### 应触发

- 询问项目为什么采用某种架构、组件或约束。
- 询问历史决策、替代方案、兼容边界或不能修改的原因。
- 排查已知问题、重复故障、常见配置错误或操作步骤。
- 执行架构设计、代码修改、缺陷修复、迁移、评审和复盘任务。
- 用户明确提到 ADR、FAQ、知识库、经验、历史结论或项目约定。

### 不应触发

- 问候、闲聊、翻译、润色和与当前项目无关的问题。
- 单纯继续上一轮、确认收到、停止或取消。
- 已提供完整事实且无需历史信息的简单操作。
- 没有 workspace 绑定的会话。

意图判定必须是本地确定性逻辑，不调用额外模型。判定结果只决定是否检索，不直接决定答案。

## 中文检索

现有按非字母数字切分的方式会把连续中文短语视为单个词项。新的知识检索词项规则为：

- 英文、数字和标识符继续使用现有规范化方式。
- 中文连续片段保留完整短语，同时生成 2 字和 3 字滑动词项。
- 标题、正文和标签使用同一词项生成器。
- 查询使用 OR 召回，但按照覆盖率、字段来源和知识类型计算相关性。
- 只包含低信息量词项的查询不触发知识注入。

代码索引继续使用现有代码检索 tokenizer，不与业务知识 tokenizer 合并。

## 排序和阈值

基础评分由以下部分组成：

- 标题命中高于正文命中。
- 标签精确命中高于正文命中。
- 查询词覆盖率参与评分，避免只命中一个常见二字词就注入。
- ADR 在架构和历史决策意图下提升权重。
- FAQ 在故障、配置和“如何处理”意图下提升权重。
- Learning 在实施、评审和复盘任务下提升权重。
- 更新时间只用于同分排序，不替代相关性。

自动上下文只保留达到最低相关性阈值的结果。主动 `knowledge_query` 可以返回低于自动注入阈值的匹配，但必须展示分数和匹配词。

## 类型化内容投影

### ADR

- 自动上下文最多 2 条。
- 保留标题、正文和来源引用。
- 单条正文上限高于 FAQ 和 Learning，避免 96 字截断破坏决策含义。
- 明确标记为历史决策参考，不允许覆盖当前任务事实。

### FAQ

- 自动上下文最多 3 条。
- 投影问题和答案，不只返回标题摘要。
- 保留来源和更新时间。

### Learning

- 自动上下文最多 3 条。
- 返回完整的短经验；超长内容按字符预算截断。
- 自动生成内容必须带来源会话和质量标记。

整个自动知识上下文使用独立字符预算。预算耗尽时优先保留高相关结果，而不是平均截断全部记录。

## 主线与任务执行

### 普通主线

构建模型消息前先执行意图判定。未触发时保持现有消息结构，不查询知识库。触发时调用统一选择器，并在当前轮用户消息前加入一个知识参考系统片段。该片段只在当前轮存在，不写回会话历史。

### 任务执行

任务标题和目标先经过相同意图判定。只有需要项目知识时才调用选择器。任务当前事实仍保持最高优先级，知识只作为参考证据。

### 主动工具查询

`knowledge_query` 继续要求 workspace 上下文，支持类型、标签和数量限制。工具返回使用统一投影，避免自动上下文和工具结果出现不同的截断、评分与字段语义。

## Workspace 绑定

- 知识能力只接受 session 已绑定的注册 workspace。
- 前端进入 workspace 或创建会话时，必须使用注册后的 workspace ID 请求知识能力。
- workspace 路径变化或 stale ID 只能通过现有注册路径解析到权威 ID，不允许创建隐式默认作用域。
- 诊断中明确区分 `unavailable`、`missing_workspace`、`not_needed`、`queried_no_match` 和 `injected`。

## 自动经验质量治理

- 自动抽取继续只生成 Learning，不自动伪造 ADR 或 FAQ。
- 保留现有严格 JSON 契约，但增加可观测失败计数。
- 写入前执行规范化精确去重和中文词项相似度去重。
- 过滤纯工具操作说明、仅描述本次任务步骤、无跨任务价值的内容。
- 同一会话单次最多写入 3 条，避免批量制造近义经验。
- 自动经验记录质量原因、来源会话和抽取时间。
- ADR 和 FAQ 仍由用户明确创建，后续若要自动建议，必须采用“建议草稿、用户确认后写入”的独立功能。

## 诊断和可观测性

每次知识能力评估生成一条结构化记录：

- `consumer`：`mainline`、`task_execution` 或 `knowledge_query_tool`。
- `decision`：`not_needed`、`missing_workspace`、`queried_no_match`、`matched_not_injected`、`injected`。
- `query_terms`、`knowledge_ids`、`knowledge_kinds`。
- `matched_count`、`injected_count`、`injected_chars`。
- `truncated` 和阈值信息。

后端已有的知识使用字段必须投影到前端 `opsView.knowledgeAudit`，使运行诊断能看到本轮是否查询、命中和注入。不得只展示“知识库可用”。

## 错误处理

- 无 workspace：跳过自动检索并记录 `missing_workspace`，不查询其他 workspace。
- 知识库为空或无匹配：继续正常对话，不注入空壳片段。
- 知识持久化失败：自动经验不得报告成功，记录可诊断错误。
- 诊断发布失败不得阻塞主对话或任务执行。
- 主动 `knowledge_query` 的错误继续通过工具结果返回明确状态。

## 测试策略

1. 知识 tokenizer：连续中文、标点中文、英文标识符和混合文本。
2. 检索排序：标题、标签、正文、类型权重、覆盖率和 workspace 隔离。
3. 意图判定：应触发和不应触发的主线及任务样例。
4. 类型化投影：ADR、FAQ、Learning 的字符预算和截断行为。
5. 主线消息：无需求时无知识查询和知识片段；有需求时只注入当前轮。
6. 任务 Prompt：无需求时 `used_knowledge=0`；有需求时记录知识 ID。
7. 自动经验：低价值过滤、近义去重、单次数量限制和失败诊断。
8. 前端投影：后端知识诊断字段能生成 `opsView.knowledgeAudit`。
9. 运行态验收：注册真实 workspace，写入 ADR/FAQ/Learning，验证三类内容在对应问题中被查询和注入。

## 验收标准

- 普通闲聊和无关任务不会产生知识查询，也不会增加知识上下文 token。
- 中文自然问句能够命中语义相关的 ADR、FAQ 或 Learning。
- 普通主线和任务执行对同一输入返回一致的知识选择结果。
- 自动注入具有稳定阈值、数量限制和字符预算。
- ADR 不再被统一压缩为 96 字摘要。
- 当前 workspace 未注册或未绑定时不会错误使用其他项目知识。
- 自动经验不会继续大量沉淀重复的工具操作说明。
- 运行诊断能够区分未需要、无匹配、命中未注入和已注入，并显示实际知识 ID。
- 相关 Rust 测试、前端检查以及真实 daemon 场景全部通过。
