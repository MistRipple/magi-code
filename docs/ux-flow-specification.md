# MultiCLI UX/UI 流程规范

> 本文档定义了主对话面板与 Worker Tab 之间的任务编排通讯交互规范

## 核心设计原则

| 原则 | 说明 |
|------|------|
| **自然语言优先** | 所有用户干预通过底部输入区域的自然语言完成 |
| **无按钮干扰** | 对话区域不显示操作按钮（Worker 卡片跳转除外） |
| **对话即控制** | 停止、重试、跳过等操作通过发送指令实现 |
| **清晰分离** | 主对话区 = 编排者叙事，Worker Tab = 执行详情 |
| **持续可交互** | 执行中仍可发送新消息，实现对话插入 |

## 双区域职责划分

| 区域 | 职责 | 内容类型 |
|------|------|---------|
| **主对话区** | 用户交互 + 编排者叙事 | 用户输入、规划说明、Worker 状态卡片、最终汇总 |
| **Worker Tab** | Worker 执行详情 | 任务说明、思考过程、工具调用、内容输出 |

---

## 消息归属与路由硬规则（不可违背）

1) **主对话区只接受编排者叙事 + Worker 节点卡片**
   - 允许：编排者分析/计划/总结/系统说明/用户消息/Worker 状态卡片（subTaskCard）
   - 禁止：Worker 直接向主对话区发送思考、工具调用、内容输出

2) **Worker Tab 只接受 Worker 自身的执行细节**
   - 允许：任务说明、思考过程、工具调用、内容输出、执行摘要
   - 禁止：编排者把执行细节写进 Worker Tab（只能下发任务说明/指令）

3) **Worker 节点卡片必须由编排者生成与更新**
   - 状态变更（执行中/等待确认/失败/完成）由编排者统一控制
   - Worker 只能上报结果，由编排者决定卡片状态变化

**Claude Tab:**

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ 📋 任务说明                                                              │
│ 分析前端项目的依赖关系，找出循环依赖和未使用的依赖                          │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 💭 思考                                                                  │
│ 我需要先读取 package.json 和 tsconfig.json 来了解项目结构...              │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🔧 read_file                                                            │
│ path: "package.json"                                                    │
│ ✅ 成功 · 1.2s                                                          │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 📝 分析结果                                                              │
│ 发现 3 个循环依赖：                                                       │
│ 1. components/A → utils/B → components/A                                │
│ 2. ...                                                                  │
└─────────────────────────────────────────────────────────────────────────┘
```

### 关键设计点

| 要点 | 说明 |
|------|------|
| 卡片布局 | 每个 Worker 独立卡片，**上下垂直排列** |
| 进度独立 | 每个 Worker 卡片有独立的状态指示 |
| 完成顺序 | 先完成的卡片先更新状态，不影响其他卡片 |
| 汇总时机 | **全部完成后**编排者才输出汇总 |

## 场景 2: Worker 依赖链执行

### 触发条件
任务之间存在依赖关系，必须顺序执行

### 主对话区流程

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ 🤖 编排者                                                               │
│ 这个任务需要分步执行：                                                    │
│ 1. Claude 先分析现有代码                                                 │
│ 2. Gemini 基于分析结果设计重构方案                                        │
│ 3. Claude 执行重构                                                       │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🟡 Claude [1/3]                                                         │
│ 分析现有代码结构                                                          │
│ 执行中...                                                    → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘

        ↓ (完成后自动触发下一步)

┌─────────────────────────────────────────────────────────────────────────┐
│ ✅ Claude [1/3]                                                         │
│ 分析现有代码结构                                                          │
│ 完成 · 识别出 5 个模块                                        → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🟡 Gemini [2/3]                                                         │
│ 设计重构方案                                                              │
│ 执行中...                                                    → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘

        ↓ (完成后继续)

┌─────────────────────────────────────────────────────────────────────────┐
│ ✅ Gemini [2/3]                                                         │
│ 设计重构方案                                                              │
│ 完成 · 生成 3 种方案                                          → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🟡 Claude [3/3]                                                         │
│ 执行重构                                                                 │
│ 执行中...                                                    → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘
```

### 关键设计点

| 要点 | 说明 |
|------|------|
| 依赖可视化 | 显示任务序号 `[1/3]` 表示执行顺序 |
| 上下文传递 | Worker B 的任务说明中包含 Worker A 的输出摘要 |
| 自动触发 | 前一个完成后自动启动下一个 |

Worker 执行过程中遇到歧义，需要用户澄清

### 主对话区流程

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ 🟡 Claude                                                               │
│ 重构用户模块                                                             │

│ 发现两种重构方案：                                                        │

│ B. Service 直连 - 简单直接，耦合较紧                                      │
│                                                                         │
│ 请告诉我你倾向哪种方案，或者让 Claude 自己决定                              │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
```

### 关键设计点

| 要点 | 说明 |
| ---- | ---- |
| 无按钮操作 | 编排者用文字描述选项，用户通过自然语言回复 |
| 状态更新 | Worker 卡片显示"等待确认"状态 |
| 上下文保持 | 用户回复后自动恢复执行 |

## 场景 4: 错误与恢复

### 触发条件
Worker 执行过程中发生错误（工具调用失败、LLM 超时等）

### 主对话区流程

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ ❌ Claude                                                               │
│ 修改配置文件                                                             │
│ 失败 · 权限不足                                              → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🤖 编排者                                                               │
│ Claude 在修改 config.json 时遇到权限问题，无法写入文件。                    │
│                                                                         │
│ 你可以：                                                                 │
│ • 说"重试"让我再次尝试                                                    │
│ • 说"跳过"继续执行后续任务                                                │
│ • 说"回滚"撤销已完成的更改                                                │
│ • 说"停止"终止整个任务                                                    │
│ • 或者告诉我其他处理方式                                                  │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 👤 用户                                                                 │
│ 跳过这个，继续后面的                                                      │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🤖 编排者                                                               │
│ 好的，已跳过配置文件修改，继续执行后续任务。                                 │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🟡 Claude                                                               │
│ 更新类型定义                                                             │
│ 执行中...                                                    → 点击跳转 │
└─────────────────────────────────────────────────────────────────────────┘
```

### 关键设计点

| 要点 | 说明 |
| ---- | ---- |
| 自然语言恢复 | 用户通过文字指令决定恢复策略 |
| 无按钮干扰 | 编排者列出可选操作但不使用按钮 |
| 状态可视化 | Worker 卡片显示失败状态 ❌ |

## UI 元素规范

### 主对话区消息卡片

| 卡片类型 | 交互方式 | 可操作元素 |
| -------- | -------- | ---------- |
| 用户消息 | 只读 | 无 |
| 编排者消息 | 只读 | 无 |
| Worker 状态卡片 | 点击跳转 | 整个卡片可点击 → 跳转到对应 Tab |
| 工具调用卡片 | 展开/折叠 | 点击内容区域展开查看详情（无按钮） |

### Worker Tab 消息卡片

| 卡片类型 | 交互方式 |
| -------- | -------- |
| 任务说明 | 只读 |
| 思考过程 | 可折叠 |
| 工具调用 | 可展开查看输入/输出 |
| 内容输出 | 只读，支持代码高亮 |
| 完成摘要 | 只读 |

### 输入区域状态

| 状态 | 输入框 | 按钮 | 行为说明 |
| ---- | ------ | ---- | -------- |
| 空闲 | 可输入 | 发送 | 正常发送消息 |
| 执行中 | **可输入** | ⏹️/发送 | 输入为空显示停止；输入有内容显示发送（补充指令） |

```text
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ 🤖 编排者                                                               │
│ 收到，我会通知 Claude 暂停对 utils 目录的修改...                          │
└─────────────────────────────────────────────────────────────────────────┘
```

**补充指令语义（明确规则）**
- 执行中新增输入属于**补充指令**：不会丢弃当前任务
- 默认不打断当前步骤，仅在**下一决策点**生效
- 如需立即打断，必须显式指令（如“停止并改为…”）

#### 输入区域按钮状态机

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                          按钮状态转换                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  空闲状态:                                                               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 输入框为空      → [发送] 按钮禁用（灰色）                          │   │
│  │ 输入框有内容    → [发送] 按钮启用                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  执行中状态:                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 输入框为空      → [⏹️ 停止] 按钮                                   │   │
│  │ 输入框有内容    → [发送] 按钮（发送后清空，按钮变回停止）            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 高频发送限制（强约束）

| 场景 | 处理方式 |
| ---- | -------- |
| 执行中高频补充 | **限频 1 秒 1 条**，超出直接提示并不发送 |
| 空闲状态高频发送 | **限频 300ms 1 条**，超出直接提示并不发送 |
| 连续超限 | 弹出提示：“发送过快，请稍后再试” |

```text
高频发送:
  消息1: 允许发送
  消息2: <1s 内发送 → 拒绝并提示
  消息3: <1s 内发送 → 拒绝并提示
```

#### 关键设计点

| 要点 | 说明 |
| ---- | ---- |
| 输入不阻塞 | 执行中用户仍可输入新消息 |
| 补充语义 | 新消息作为补充指令，不丢弃当前任务 |
| 上下文感知 | 编排者理解这是执行中的追加指令，必要时调整 Worker 行为 |
| 按钮双态 | 输入框为空显示停止，有内容显示发送 |
| 限频规则 | 超频直接拒绝并提示，不进入队列 |

## 术语表（Glossary）

| 术语 | 定义 |
| ---- | ---- |
| **编排者（Orchestrator）** | 负责理解用户意图、分配任务、协调 Worker 的核心 AI。所有 Worker 间通讯均通过编排者中转 |
| **Worker** | 执行具体任务的 AI Agent（如 Claude/Codex/Gemini），只负责执行，不直接与其他 Worker 通讯 |
| **主对话区** | 用户与编排者交互的主界面，展示高层叙事（规划、状态卡片、汇总） |
| **Worker Tab** | 每个 Worker 独立的执行详情面板，展示思考、工具调用、内容输出等 |
| **Worker 状态卡片** | 主对话区展示的 Worker 执行状态摘要卡片，可点击跳转到对应 Tab |
| **任务说明（Instruction）** | 编排者派发给 Worker 的任务描述，显示在 Worker Tab 顶部 |
| **补充指令** | 执行中用户发送的追加消息，不中断当前任务，在下一决策点生效 |
| **依赖链执行** | 任务之间存在依赖关系，必须按顺序执行（A→B→C） |
| **并行执行** | 多个 Worker 同时执行无依赖关系的任务（A\|\|B） |
| **上下文传递** | 编排者摘要前序 Worker 输出，注入后续 Worker 的任务说明 |
| **限频** | 防止高频发送的速率限制机制（执行中 1秒/条，空闲 300ms/条） |
