## 当前项目状态

`magi-rust-rewrite` 当前不是独立新产品线，也不是只做最小后端验证的临时仓。

当前真实定位是：

- 以 Rust 为核心完成后端重构与协议冻结
- 在冻结契约基础上推进现有 `magi` 前端方案的正式迁移
- 保留现有产品层方案，同时系统性重构前端接线层、状态层、投影层与宿主桥接边界

也就是说：

- Rust 重构的是后端能力与契约边界
- 前端不是重写一个全新产品
- 前端也不是长期停留在“最小客户端 + 协议验证”状态
- 当前仓内已经允许并需要保留正式前端迁移所需的页面、组件树、状态分层与 UI 表达

## 当前阶段约束

当前阶段仍然遵守 `rust-backend-rewrite/01-governance-and-rules.md` 与
`rust-backend-rewrite/05-milestones-and-cutover-gates.md` 的核心原则，但约束重点已经调整为：

- 协议字段、DTO 结构、事件信封与宿主桥接命令必须继续保持冻结与收敛
- 前端迁移必须以现有 `magi` 产品方案为基线，不发散成新的 UI 产品方向
- 页面层、组件层、store 分层与 projection 模型可以继续建设，但必须服务于正式前端迁移，不得形成长期双实现
- 禁止用兼容分支、回退逻辑、补丁式过渡掩盖结构问题
- UI 层可以继续完善正式产品表达，但不能反向定义后端协议

## 当前前端迁移完成面

当前仓内前端迁移已经完成的主干包括：

- 正式工作台壳层已经建立，并由 `WebWorkbenchShell -> App` 驱动主界面装配
- 页面主链已切到 `contract-store + projection-store`，`runtime-store` 已收缩为薄门面
- 主线程区、任务区、变更区、知识区、设置区等正式面板已经建立并接入当前读模型
- `ThreadPanel -> MessageList -> MessageItem` 的消息链路已经收束到统一读模型入口
- 消息内容已统一收束为 `TimelineRenderItem.content` 结构，并开始区分普通文本与结构化正文显示
- 前端文案、术语与产品表达已经过多轮收口，当前整体方向已接近正式产品前端表达

当前前端迁移的重点已经不再是“是否允许保留页面与组件树”，而是：

- 继续保持产品层方案与正式体验
- 在冻结契约下稳步完善接线层与投影层
- 为后续真实能力接入准备稳定的页面结构、消息读模型和宿主桥接边界

## 前端迁移实施原则与下一阶段方向

前端迁移继续遵守以下原则：

- 以现有 `magi` 产品方案为基线迁移，不发散式重写新前端
- 先收敛契约与读模型，再推进页面与组件层表现
- 页面可以继续正式化，但不能在 UI 层伪造后端已完成能力
- 所有接线和状态整理应优先沉到契约层、投影层与消息读模型层，而不是堆在页面组件里
- 禁止长期双实现、回退逻辑与补丁式兼容分支

下一阶段方向主要包括：

- 在现有页面结构稳定的基础上，继续准备真实能力接入
- 继续围绕消息渲染、任务树、知识结果、变更预览、设置数据等读模型完善契约边界
- 在不破坏当前产品层表达的前提下，把“迁移收口”平滑过渡到“真实能力接入准备”

## 当前不允许的方向

- 把本仓前端再次收回到“只允许最小客户端”的旧限制
- 脱离现有 `magi` 产品方案，发散式重写一套新的 Rust 风格前端
- 为了短期跑通而保留长期双实现、双真相源或回退逻辑
- 在 UI 层伪造后端能力完成度，掩盖契约尚未补齐的事实
