# MultiCLI 项目优势

## 核心价值主张

MultiCLI 是一个**编排者主导的多 AI CLI 协同工作平台**，通过统一的消息流架构和 Worker 画像系统，实现了多个 AI CLI 工具的高效协同，为复杂软件工程任务提供了工业级解决方案。

---

## 核心优势

### 编排与协同

**编排者主导的工作流**

单一协调者负责任务的规划、分发、集成和总结，确保整个工作流的一致性和可控性。编排者作为中央控制节点，统筹全局视角，避免了多 Agent 系统中常见的协调混乱问题。

**多 CLI 协同执行**

支持 Claude、Codex、Gemini 等多个 AI CLI 工具的并行或串行执行。系统根据任务特性和 Worker 能力，智能分配任务给最合适的执行者，充分发挥各 CLI 的专长。

**依赖感知调度**

基于 DAG（有向无环图）的任务调度引擎，自动分析任务间的依赖关系。无依赖的任务并行执行以提升效率，有依赖的任务按序执行以保证正确性。

**功能契约与验收标准**

在任务执行前明确定义功能契约和验收标准，确保前后端、不同模块之间的接口对齐，减少集成问题。

### 安全与可靠性

**文件级锁定机制**

实现细粒度的文件级锁定，防止多个 Worker 同时修改同一文件导致的冲突。系统自动检测文件依赖关系，避免竞态条件。

**快照与回滚支持**

每次文件修改自动创建快照，记录完整的变更历史。支持任务级别的回滚操作，快照元数据包含 Worker 信息和任务上下文，便于问题追溯。

**内置验证阶段**

集成编译、Lint、测试等验证钩子作为质量门禁。任务完成后自动触发验证流程，确保代码质量符合标准。

**审查循环机制**

在最终集成前进行自检和互检，Worker 之间可以相互审查代码变更，提供反馈和改进建议，形成质量保障的闭环。

### 智能与效率

**上下文压缩技术**

在长会话中智能压缩上下文，保留关键信息的同时节省 Token 消耗。通过语义分析识别重要内容，丢弃冗余信息。

**执行统计驱动优化**

基于历史执行数据动态调整 Worker 选择策略。系统自动学习哪个 Worker 更适合哪类任务，持续优化任务分配决策。

**三层上下文管理**

- **Session Context**: 会话级上下文，跨任务共享
- **Task Context**: 任务级上下文，单个任务内共享
- **Worker Context**: Worker 级上下文，特定执行者的专属信息

Memory 系统提供跨任务的知识传递，避免重复工作。

### 可观测性

**完整的任务追踪**

提供完整的任务时间线、输出和工具调用面板。用户可以清晰了解每个任务的执行过程、耗时、资源消耗等关键指标。

**端到端消息追踪**

从消息产生到最终渲染的全链路追踪，记录消息在各层的流转过程。支持消息重放和调试，快速定位问题根源。

**实时进度反馈**

流式输出与任务状态实时更新，用户可以随时了解任务执行进度。支持大文件输出的增量渲染，避免 UI 阻塞。

## 统一消息流架构

### 架构设计

MultiCLI 采用**统一消息流架构**，从产品架构层面解决多 CLI 协同的核心挑战。

#### 设计原则

不同 CLI 工具的输出格式完全不同：Claude 和 Gemini 使用 Stream JSON 格式，Codex 使用 Plain Text 格式。传统方案针对每个 CLI 编写特殊处理逻辑，导致代码混乱、难以维护。

MultiCLI 的解决方案是建立统一的消息流处理管道：

```
Raw Output (各 CLI 原始输出)
     ↓
CLI Parser (针对性解析器)
  - ClaudeParser: 处理 Stream JSON
  - CodexParser: 处理 Plain Text
  - GeminiParser: 处理 Stream JSON
     ↓
UnifiedMessage (统一消息模型)
  - 单一数据结构贯穿全系统
  - 包含完整的元数据和渲染指令
     ↓
Message Transformer (消息转换器)
  - RenderingRuleEngine: 集中式渲染决策
  - 根据消息类型、来源、上下文决定渲染方式
     ↓
Message Router (消息路由器)
  - 根据渲染指令分发到不同处理器
     ↓
UI Bridge (UI 桥接层)
  - 转换为 Webview 协议
     ↓
Webview (用户界面)
  - 对话面板 / CLI 面板 / 工具调用面板
```

### 核心组件

#### 单一数据模型

所有消息都使用统一的 UnifiedMessage 结构：

```typescript
interface UnifiedMessage {
  id: string;
  timestamp: number;
  source: MessageSource;      // 消息来源
  content: MessageContent;    // 消息内容
  context: MessageContext;    // 上下文信息
  rendering: RenderingDirective; // 渲染指令
}
```

**价值**：
- 消息处理逻辑清晰，减少 bug
- 新增 CLI 只需添加 Parser，无需修改下游逻辑
- 便于调试和追踪消息流

#### 集中式渲染决策

RenderingRuleEngine 集中管理所有渲染规则：

```typescript
class RenderingRuleEngine {
  decide(message: UnifiedMessage): RenderingDirective {
    // 根据消息类型、来源、上下文统一决策
    // 避免渲染逻辑分散在各处
  }
}
```

**价值**：
- 渲染行为一致可预测
- 修改渲染规则只需改一处
- 支持复杂的渲染策略（流式输出、折叠、优先级）

#### 可扩展的 Parser 工厂

每个 CLI 有独立的 Parser：

```typescript
class CLIParserFactory {
  getParser(cliType: 'claude' | 'codex' | 'gemini'): CLIParser {
    // 返回对应的 Parser
  }
}
```

**价值**：
- 新增 CLI 支持简单（只需实现 Parser 接口）
- 各 CLI 的解析逻辑互不干扰
- 便于针对性优化和调试

#### 端到端消息追踪

MessageTracer 提供完整的消息追踪：

```typescript
class MessageTracer {
  trace(message: UnifiedMessage): void {
    // 记录消息在各层的流转
    // 便于调试和性能分析
  }
}
```

**价值**：
- 问题定位快速准确
- 性能瓶颈一目了然
- 支持消息重放和调试

## Worker 画像系统

### 设计理念

MultiCLI 采用独特的 **Worker Profile System**，通过配置化的方式定义 Worker 的能力、偏好和行为引导。

#### 核心洞察

Worker 是完整的 CLI 进程（Claude Code / Codex / Gemini CLI），它们自带：
- 完整的工具集（文件读写、命令执行、代码分析等）
- 上下文管理能力
- 代码编辑能力
- 错误恢复机制

因此，MultiCLI 不需要限制工具权限（CLI 自己管理），不需要实现复杂的 Agent 角色，只需要引导 Worker 专注于特定领域。

### 核心特性

#### CLI 进程复用

不重复造轮子，利用成熟 CLI 的完整能力。Claude Code、Codex、Gemini CLI 都是经过验证的工具，MultiCLI 直接复用它们，自动获得所有更新和改进。

#### 引导而非限制

通过 Prompt 注入引导 Worker 专注于特定领域，不限制工具权限，保持 CLI 的完整能力。这种方式更灵活，Worker 可以根据实际情况调整行为。

#### 执行统计驱动

基于历史执行数据动态调整 Worker 选择。系统自动学习哪个 Worker 更适合哪类任务，持续优化任务分配策略。

#### 三层上下文管理

- **Session Context**: 会话级上下文
- **Task Context**: 任务级上下文
- **Worker Context**: Worker 级上下文
- **Memory 系统**: 提供跨任务的知识传递

#### 纯配置扩展

添加新 Worker 只需添加配置文件，调整职责只需修改 YAML，无需编写代码，降低使用门槛。

### 配置文件结构

```
.multicli/
├── config.json              # 全局配置
├── workers/                 # Worker 画像配置
│   ├── claude.yaml          # Claude Worker 画像
│   ├── codex.yaml           # Codex Worker 画像
│   ├── gemini.yaml          # Gemini Worker 画像
│   └── custom/              # 自定义画像（可选）
├── categories/              # 任务分类配置
│   └── default.yaml         # 默认分类规则
└── memory/                  # 会话记忆
```

### Worker 画像配置示例

```yaml
# .multicli/workers/claude.yaml
name: claude
displayName: Claude

# 能力画像（用于任务分配决策）
profile:
  strengths:
    - 复杂架构设计
    - 代码重构
    - 深度推理
  costFactor: 1.0
  speedFactor: 0.8
  qualityFactor: 1.2

# 任务偏好
preferences:
  preferredCategories:
    - architecture
    - refactor
    - integration

# 行为引导（注入到 Worker Prompt）
guidance:
  role: |
    你是一个资深软件架构师，专注于系统设计和代码质量。
  focus:
    - 优先考虑代码的可维护性和扩展性
    - 在修改前先分析影响范围
  constraints:
    - 不要进行不必要的重构
    - 避免引入新的依赖
```

## 技术实现亮点

### 消息流处理

**流式输出支持**

- 实时显示 Worker 执行进度
- 支持大文件输出的增量渲染
- 避免 UI 阻塞，提升用户体验

**消息批处理**

- 合并频繁的小消息，减少 UI 更新开销
- 智能缓冲策略，平衡实时性和性能

### 依赖图调度

**DAG 执行引擎**

- 自动分析任务依赖关系
- 并行执行无依赖任务，提升效率
- 串行执行有依赖任务，保证正确性

**执行示例**

```
任务 A: 设计数据库 Schema
任务 B: 实现后端 API（依赖 A）
任务 C: 实现前端 UI（依赖 B）
任务 D: 编写测试用例（依赖 B）

执行顺序：A → B → (C 和 D 并行)
```

### 文件锁定机制

**冲突预防**

- 同一文件同时只能被一个 Worker 修改
- 自动检测文件依赖，避免竞态条件
- 支持文件级别的细粒度锁定

### 快照与回滚

**版本管理**

- 每次文件修改自动创建快照
- 支持任务级别的回滚
- 快照元数据包含 Worker 信息和任务上下文

## 总结

MultiCLI 通过**统一消息流架构**和**Worker 画像系统**两大核心设计，从产品架构层面解决了多 AI CLI 协同的根本问题：

1. **统一消息流架构**：从根源上解决不同 CLI 输出格式不一致的问题，提供清晰的消息处理流程
2. **Worker 画像系统**：通过引导而非限制的方式，充分利用成熟 CLI 的完整能力
3. **编排者主导**：单一协调者负责任务规划、分发、集成和总结，确保工作流的一致性
4. **依赖感知调度**：基于 DAG 的执行引擎，自动优化任务执行顺序
5. **完整可观测性**：端到端的消息追踪和 UI 展示，问题定位快速准确

这些设计使 MultiCLI 成为一个**工业级的多 AI CLI 协同工作平台**，为复杂软件工程任务提供了高效、可靠、易用的解决方案。
