# MultiCLI 文件快照系统架构深度分析

## 概述

本文档从产品视角分析 MultiCLI 文件快照系统的设计合理性,评估其是否为最优方案,并提出架构改进建议。

---

## 1. 当前架构剖析

### 1.1 核心组件

**SnapshotManager** (src/snapshot-manager.ts)
- 职责: 文件快照的创建、存储、还原
- 存储: `.multicli/sessions/{sessionId}/snapshots/{snapshotId}.snapshot`
- 元数据: 存储在 UnifiedSessionManager 的 session.snapshots

**FileLockManager** (src/orchestrator/file-lock-manager.ts)
- 职责: 文件级排他锁,防止并发写冲突
- 机制: 优先级队列 + 饥饿避免 (15s boost)
- 范围: 内存级别,进程退出即失效

**RecoveryHandler** (src/orchestrator/recovery-handler.ts)
- 职责: 基于快照的失败恢复和回滚
- 策略: retry → escalate → rollback

**TaskDependencyGraph** (src/orchestrator/task-dependency-graph.ts)
- 职责: 文件冲突检测,串行化冲突任务
- 机制: DAG 分析 + 文件依赖边

### 1.2 数据流

```
Worker 修改文件前:
  └─> SnapshotManager.createSnapshot()
      ├─> 读取文件原始内容
      ├─> 保存到 {sessionId}/snapshots/{snapshotId}.snapshot
      └─> 元数据存入 session.snapshots[]

并发任务调度:
  └─> TaskDependencyGraph.addFileDependencies()
      ├─> 检测文件冲突
      ├─> 添加串行依赖边
      └─> 分批执行避免冲突

任务执行期间:
  └─> FileLockManager.acquire(files)
      ├─> 检查文件锁状态
      ├─> 冲突则进入优先级队列
      └─> 获取锁后执行,完成后释放

任务失败时:
  └─> RecoveryHandler.performRollback()
      └─> SnapshotManager.revertToSnapshot()
          └─> 从 .snapshot 文件还原原始内容
```

## 2. 架构问题分析

### 2.1 🔴 P0 级问题 (严重架构缺陷)

#### 问题 1: 快照与文件锁机制冗余且不协调

**现状**:
- **TaskDependencyGraph** 在任务调度前检测文件冲突,串行化任务
- **FileLockManager** 在任务执行时动态加锁,防止并发

**问题**:
```typescript
// TaskDependencyGraph 已经确保串行化
graph.addFileDependencies('sequential'); // 冲突任务串行

// 但 FileLockManager 仍然在运行时加锁
const release = await fileLockManager.acquire(['src/file.ts']); // 冗余!
```

**后果**:
1. **双重开销**: 调度期检测 + 运行时加锁
2. **可能死锁**: DAG 未考虑锁的优先级,可能出现循环等待
3. **逻辑混乱**: 两个组件都在解决"文件冲突"问题,职责不清

**根因**: TaskDependencyGraph 是静态分析,FileLockManager 是动态保护。但如果静态分析已经保证串行,动态锁就是冗余的。

---

#### 问题 2: 快照粒度过粗,缺乏版本管理

**现状**:
```typescript
// snapshot-manager.ts:67-84
const existingSnapshot = session.snapshots.find(
  s => s.filePath === relativePath && s.subTaskId === subTaskId
);
if (existingSnapshot) {
  // 同一 SubTask 重复创建快照,直接返回现有快照
  return { ...existingSnapshot, originalContent };
}
```

**问题**:
- **单版本快照**: 每个文件只保存一次初始状态,无中间版本
- **无法细粒度回滚**: 如果 Task1 成功 → Task2 失败,只能回滚到 Task1 之前的状态
- **多任务冲突**: 不同 SubTask 修改同一文件,只保留第一个快照

**场景示例**:
```
初始状态: file.ts (version 0)
Task1 执行: file.ts (version 1) ← 创建快照
Task2 执行: file.ts (version 2) ← 无快照,复用 Task1 的
Task3 执行: file.ts (version 3) ← 无快照,复用 Task1 的

Task3 失败回滚 → 直接回到 version 0,丢失 Task1/Task2 的修改!
```

**后果**:
- **粒度回滚不可行**: 只能全部回滚或全部保留
- **依赖链断裂**: Task3 依赖 Task2 的修改,回滚后破坏依赖
- **用户体验差**: 无法只回滚失败的部分,保留成功的部分

---

#### 问题 3: 快照存储与 Git 冲突,数据安全隐患

**现状**:
```
.multicli/
  └─ sessions/
      └─ {sessionId}/
          ├─ snapshots/
          │   ├─ {snapshotId1}.snapshot  ← 完整文件内容
          │   ├─ {snapshotId2}.snapshot
          │   └─ {snapshotId3}.snapshot
          └─ session.json
```

**问题**:
1. **与 Git 重复**: Git 本身就是版本管理系统,快照实际上是重新实现了 Git Staging Area
2. **无版本历史**: 只保存单一快照,无法查看完整修改历史
3. **冲突风险**: 快照回滚可能与 Git 状态不一致
   - 快照: 回到 version 0
   - Git working tree: 仍然是 version 3
   - Git staging area: 可能已经 add 了 version 2
4. **存储浪费**: 重复保存完整文件内容,而 Git 已有增量存储

**安全隐患示例**:
```bash
# 用户场景
1. Task1 修改 file.ts → 快照保存 version 0
2. 用户手动 git add file.ts → Git staging 保存 version 1
3. Task2 修改 file.ts → 文件变为 version 2
4. Task2 失败,快照回滚到 version 0
5. 用户 git commit → Git 提交的是 version 1 (来自 staging)

结果: 代码仓库中的 version 1 != 工作区的 version 0
```

---

#### 问题 4: 快照创建时机不一致,覆盖检测缺失

**现状**:
```typescript
// worker-agent.ts:284-290
if (response.fileChanges && response.fileChanges.length > 0 && this.snapshotManager) {
  for (const change of response.fileChanges) {
    try {
      this.snapshotManager.createSnapshot(change.filePath, this.type, subTask.id);
    } catch (err) {
      console.warn(`[WorkerAgent] 创建快照失败: ${change.filePath}`, err);
    }
  }
}
```

**问题**:
- **事后快照**: 在 Worker **执行后**根据 `response.fileChanges` 创建快照
- **无法捕获实际修改**: 依赖 LLM 返回的 fileChanges,可能遗漏或错误
- **无覆盖检测**: 如果 Worker 实际修改的文件 != fileChanges,无法检测

**风险示例**:
```typescript
// LLM 返回
response.fileChanges = ['src/types.ts'];

// 但实际修改了
- src/types.ts
- src/utils.ts  ← 未在 fileChanges 中,无快照!
- src/index.ts  ← 未在 fileChanges 中,无快照!

// 任务失败回滚 → types.ts 回滚,但 utils.ts 和 index.ts 仍然被修改
```

**根因**: 快照依赖 LLM 声明,而非文件系统监控

### 2.2 🟡 P1 级问题 (设计不合理)

#### 问题 5: 快照清理策略不明确

**现状**:
```typescript
// snapshot-manager.ts:358-364
cleanupSession(sessionId: string): void {
  const snapshotDir = this.getSnapshotDir(sessionId);
  if (fs.existsSync(snapshotDir)) {
    fs.rmSync(snapshotDir, { recursive: true, force: true });
  }
}
```

**问题**:
- **手动清理**: 依赖调用 cleanupSession(),未自动清理
- **无 TTL**: 快照文件永久保留,占用磁盘
- **无 LRU**: 长期会话会积累大量快照

**后果**: 磁盘占用无限增长

---

#### 问题 6: 快照内容 diff 算法过于简单

**现状**:
```typescript
// snapshot-manager.ts:264-281

  const originalLines = original.split('\n');
  const currentLines = current.split('\n');

  const additions = Math.max(0, currentLines.length - originalLines.length);
  const deletions = Math.max(0, originalLines.length - currentLines.length);
  // ...

```

**问题**:
- **行数 diff**: 只比较总行数,无法精确定位变更
- **误报**: 插入空行也算 addition
- **无语义 diff**: 无法识别代码结构变更 (函数移动、重构等)

**改进空间**: 使用 git diff 或 LCS (Longest Common Subsequence) 算法

### 2.3 🟢 P2 级问题 (可优化)

#### 问题 7: 快照元数据存储效率低

**现状**: 快照元数据存在 session.json,每次修改需要序列化整个 session

**优化**: 使用 SQLite 或独立的 snapshots.json

---

#### 问题 8: 无快照压缩机制

**问题**: 大文件快照占用空间大 (如 package-lock.json, bundle.js)

**优化**: 使用 gzip 压缩快照文件

### 3.2 与业界方案对比

#### Git-based Approach (推荐)

**方案**: 直接使用 Git 作为快照系统

```typescript
class GitSnapshotManager {
  async createSnapshot(files: string[]): Promise<string> {
    // 1. 创建临时 stash
    await git.stash.push({ files, message: `snapshot-${Date.now()}` });
    return stashId;
  }

  async revertToSnapshot(stashId: string): Promise<void> {
    // 2. 应用 stash
    await git.stash.apply(stashId);
  }

  async acceptChanges(files: string[]): Promise<void> {
    // 3. 加入 staging area
    await git.add(files);
  }
}
```

**优势**:
- ✅ 细粒度版本管理 (每个 stash 是一个版本)
- ✅ 增量存储 (Git blob storage)
- ✅ 与 Git 工作流一致
- ✅ 支持 diff, log, cherry-pick
- ✅ 无需自己实现存储逻辑

**劣势**:
- ⚠️ 依赖 Git 仓库 (非 Git 项目无法使用)
- ⚠️ Stash 栈有限制

---

#### Copy-on-Write (COW) Approach

**方案**: 使用文件系统 COW 特性 (Btrfs, ZFS)

```typescript
class COWSnapshotManager {
  async createSnapshot(files: string[]): Promise<string> {

```

**优势**:
- ✅ 零拷贝,性能极高
- ✅ 自动去重 (相同内容共享存储)

**劣势**:
- ❌ 依赖特定文件系统
- ❌ 跨平台兼容性差

#### Database-backed Versioning

**方案**: 使用 SQLite 存储文件版本链

```typescript
class DBSnapshotManager {
  async createSnapshot(file: string): Promise<number> {
    // 1. 读取当前内容
    const content = await fs.readFile(file);
    // 2. 插入新版本
    const version = await db.exec(`
      INSERT INTO file_versions (file_path, content, parent_version)
      VALUES (?, ?, ?)
    `, [file, content, latestVersion]);
    return version;
  }

  async revertToVersion(file: string, version: number): Promise<void> {
    // 3. 回滚到指定版本
    const content = await db.get('SELECT content FROM file_versions WHERE version = ?', version);
    await fs.writeFile(file, content);
  }
}
```

**优势**:
- ✅ 细粒度版本管理
- ✅ 支持版本树 (分支、合并)
- ✅ 高效查询 (SQL)

**劣势**:
- ⚠️ 存储开销 (完整内容存储)
- ⚠️ 需要维护数据库

#### 优化 1: 移除 FileLockManager,统一到 TaskDependencyGraph

```typescript
  const watcher = new FileWatcher(subTask.targetFiles);

    console.log(`[Worker] 文件被修改: ${file}`);

  const response = await this.execute(subTask);

    console.warn(`  声明: ${declaredChanges.join(', ')}`);
    console.warn(`  实际: ${actualChanges.join(', ')}`);

  watcher.stop();
}
```

### 4.2 中期重构 (1-2月,改架构)

#### 方案: 基于 Git 的快照系统

**架构**:
```typescript
class GitSnapshotManager implements SnapshotManager {
  private git: SimpleGit;

  async createSnapshot(files: string[], modifiedBy: CLIType, subTaskId: string): Promise<string> {
    // 1. 检查是否为 Git 仓库
    if (!await this.git.checkIsRepo()) {
      // 降级到文件快照
      return super.createSnapshot(files, modifiedBy, subTaskId);
    }

    // 2. 创建 Git stash (带元数据)
    const stashMessage = JSON.stringify({
      subTaskId,
      modifiedBy,
      timestamp: Date.now(),
      files,
    });

    await this.git.stash(['push', '-m', stashMessage, '--', ...files]);

    // 3. 获取 stash ref
    const stashRef = await this.git.raw(['rev-parse', 'stash@{0}']);
    return stashRef.trim();
  }

  async revertToSnapshot(stashRef: string): Promise<void> {
    // 应用 stash (不删除)
    await this.git.stash(['apply', stashRef]);
  }

  async acceptChanges(files: string[]): Promise<void> {
    // 加入 Git staging area
    await this.git.add(files);
  }

  async getVersionHistory(file: string): Promise<GitVersion[]> {
    // 使用 git log 获取文件历史
    const log = await this.git.log({ file });
    return log.all.map(commit => ({
      hash: commit.hash,
      message: commit.message,
      author: commit.author_name,
      date: new Date(commit.date),
    }));
  }
}
```

**优势**:
- ✅ 无需自己实现存储
- ✅ 与 Git 工作流一致
- ✅ 支持版本历史、diff、blame
- ✅ 增量存储,空间高效

**迁移策略**:
1. 保留原 SnapshotManager 作为 fallback (非 Git 项目)
2. 新增 GitSnapshotManager 作为 Git 项目的默认实现
3. 自动检测是否为 Git 仓库,选择实现

## 5. 总结与建议

### 5.1 当前架构评价

**优点**:
- ✅ 基础功能完整 (快照、回滚、冲突检测)

- ❌ 存储效率低 (完整内容存储,无压缩,无 TTL)

**是否为最优方案?** → **NO**

```typescript
  async listSnapshots(): Promise<{ ref: string; message: string; date: Date }[]> {
    const stashList = await this.git.stashList();
    return stashList.all.map(stash => ({
      ref: stash.hash,
      message: stash.message,
      date: new Date(stash.date),
    }));
  }

  async diff(stashRef: string): Promise<string> {
    return await this.git.show([stashRef]);
  }
}
```

### 6.2 文件监控器实现

```typescript
import * as chokidar from 'chokidar';

export class FileWatcher {
  private watcher: chokidar.FSWatcher;
  private changedFiles = new Set<string>();

  constructor(files: string[]) {
    this.watcher = chokidar.watch(files, {
      persistent: false,
      ignoreInitial: true,
    });

    this.watcher.on('change', (path) => {
      this.changedFiles.add(path);
    });
  }

  getChangedFiles(): string[] {
    return Array.from(this.changedFiles);
  }

  stop(): void {
    this.watcher.close();
  }
}
```
