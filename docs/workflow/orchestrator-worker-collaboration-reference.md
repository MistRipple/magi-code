# Magi 编排者-Worker 协作设计参考

## 1. 文档目的

定义 Magi 作为 AI 编排平台的核心协作模型，解决五个基本问题：

- 编排者和 Worker 各自负责什么，决策权如何分配
- 任务如何从用户需求变成可交付结果
- 多 Worker 并行时如何协调
- 用户在流程中扮演什么角色
- 出现异常时如何恢复而非阻塞

---

## 2. 角色定位

### 2.1 编排者（Orchestrator）

**定位：目标管理者 + 协作协调者。**

编排者的核心能力是**将模糊的用户意图转化为可验收的任务合同**，并在多 Worker 协作时维护契约一致性。

职责边界：

| 必须做 | 不应做 |
| ------ | ------ |
| 理解用户意图，拆分为独立任务 | 指定具体实现步骤 |
| 定义每个任务的目标和验收标准 | 预设算法选择或代码结构 |
| 声明跨任务协作契约 | 把 Worker 降级为指令执行器 |
| 评估交付结果是否达标 | 替 Worker 做技术决策 |
| 处理失败恢复和任务改派 | — |

**关于任务拆分的粒度**：编排者按业务目标拆分（"实现登录功能"、"修复支付 bug"），不按实现步骤拆分（"先改数据库，再改接口，再改前端"）。如果编排者无法判断某个需求应该拆成几个任务，应倾向于少拆——让 Worker 在执行中自行处理内部复杂性，好过编排者基于不充分信息做错误拆分。

### 2.2 Worker

**定位：自主工程角色。**

Worker 对其任务的技术决策拥有完全自主权，包括：

1. **分析**：理解现状、评估约束、识别风险
2. **决策**：选择实现方案并说明理由
3. **执行**：完成改动并自行验证
4. **上报**：遇到无法独立解决的问题时通知编排者

Worker 不需要为每个技术决策请求编排者批准。"自主"意味着在合同范围内，Worker 拥有等同于人类工程师的决策权。

## 3. 任务合同模型

分配给 Worker 的内容是"合同"，不是"施工图"。

### 3.1 合同字段

| 字段 | 必填 | 说明 |
| ---- | ---- | ---- |
| 目标（Goal） | 是 | 要达成的业务结果 |
| 验收标准（Acceptance） | 是 | 明确的完成判定条件 |
| 约束（Constraints） | 是 | 必须遵守的规则和禁止行为 |
| 上下文（Context） | 是 | 编排者已掌握的事实、线索、相关信息 |
| 范围线索（Scope Hint） | 建议 | 编排者预判的初始工作范围（文件/模块），非硬约束 |
| 协作契约（Contracts） | 条件必填 | 仅当多 Worker 协作时：接口定义、数据格式、依赖关系 |

**关于范围线索**：`scope_hint` 是编排者对工作范围的初始预判，用于支撑并行任务的文件级分区（见 8.1），**不是硬性约束**。Worker 可以超出此范围工作——超出时无需审批，但实际改动范围会在 Phase C 审计中被检查（见 7.3）。如果编排者无法预判范围，可以省略此字段，Worker 自行确定。

### 3.2 合同中禁止的内容

- 逐步实现脚本："先改 A，再改 B，再运行 C"
- 强制技术选型："必须用某函数名/某设计模式"
- 伪装成约束的微操："只允许修改 file.ts 的第 30-50 行"

**例外**：安全合规要求、跨 Worker 接口契约、用户明确强制的规范。

## 4. 任务分级与判定

不是所有任务都需要同等程度的自主性。过度授权浪费资源，不足授权限制质量。

### 4.1 分级定义

| 级别 | 特征 | 编排策略 |
| ---- | ---- | -------- |
| **L1 轻量任务** | 范围明确、改动局部、无跨模块影响 | 简洁合同，Worker 直接执行 |
| **L2 标准任务** | 需要分析和方案选择，可能跨模块 | 完整合同，Worker 自主决策 |
| **L3 复杂任务** | 跨多模块/多 Worker 协作、有架构影响 | 完整合同 + 协作契约，编排者主动协调 |

### 4.2 判定规则

编排者根据**用户意图的结构特征**判定级别，而非根据实现复杂度（实现复杂度是 Worker 分析后才能确定的）。

判定依据：

| 信号 | L1 | L2 | L3 |
| ---- | -- | -- | -- |
| 用户需求涉及几个独立关注点 | 1 个 | 1 个 | 多个 |
| 是否涉及多个技术领域（前端/后端/数据等） | 否 | 可能 | 是 |
| 是否需要多个 Worker 协作 | 否 | 否 | 是 |
| 用户是否提供了明确的改动范围 | 是 | 不确定 | 不确定 |
| 是否有跨模块接口影响 | 否 | 可能 | 是 |

**判定原则**：

- **不确定时向下兼容**：拿不准 L1 还是 L2 时按 L2 处理，拿不准 L2 还是 L3 时按 L3 处理。宁可多给上下文和契约，不可给出模糊的半成品合同。
- **分级不是一次性的**：Phase C 汇总时如果发现任务实际复杂度超出初始分级，下一轮编排应调整。

## 5. 安全与边界

### 5.1 硬边界（系统强制）

由工具层和运行时强制执行，Worker 无法绕过，不需要协商：

- 工作区外文件系统访问
- 密钥/凭据操作
- 破坏性系统命令

设计原则：**通过能力限制实现，而非通过提示词约束**。Worker 不是"被告知不能做"，而是"物理上做不到"。

### 5.2 软边界（分类治理）

Worker 在执行过程中可能需要访问合同未预期的代码范围。按操作的可逆性采取不同策略：

**可逆操作**（文件读写、代码修改）：

- **默认放行**：Worker 在工作区内自由读写
- **事后审计**：编排者在 Phase C 检查实际改动范围
- 审计标准见 7.3 节

**不可逆操作**（外部 API 调用、数据库写入、发布部署）：

- **声明制**：Worker 在执行不可逆操作前，必须在结果报告中声明意图和预期影响
- **工具层拦截**：高危不可逆操作由工具权限系统控制，Worker 无法静默执行
- 本质上与硬边界的区别是：硬边界完全禁止，不可逆操作是**有条件允许**

### 5.3 协作边界（同步约束）

当多个 Worker 并行工作时，跨任务的契约变更需要编排者协调（详见第 8 节）。

---

## 6. 编排者约束权限

编排者可以对 Worker 施加强制约束，但仅限以下类别：

| 可强制 | 不可强制 |
| ------ | -------- |
| 安全与权限规则 | 具体实现步骤 |
| 跨 Worker 协作契约 | 代码写法和技术选型 |
| 验收标准 | 内部工作范围 |
| 用户明确要求的规范 | 执行顺序（除非有依赖） |

---

## 7. 协作闭环

### 7.1 Phase A：规划与分配

**输入**：用户需求 + 项目上下文

**编排者动作**：

1. 理解用户意图，判断任务分级（参照 4.2 判定规则）
2. 按业务目标拆分任务，为每个任务定义合同
3. 根据任务分类（category）路由到对应 Worker
4. 声明任务间关系：并发（无依赖）或串行（有依赖）
5. 对 L3 任务：定义跨 Worker 协作契约

**输出**：一组结构化任务合同 + 依赖拓扑

**信息传递规则**：

- 每个任务合同必须自包含——Worker 不应需要参照其他任务的合同才能理解自己的任务
- 共享上下文（如项目背景、用户偏好）应复制到每个合同的上下文字段中，而非通过引用
- 多 Worker 并行任务应在合同中显式提供 `scope_hint`，并尽量保证不同任务的 `scope_hint` 不重叠；`scope_hint` 仅用于引导聚焦，不构成硬约束

### 7.2 Phase B：并行执行

**输入**：Phase A 产出的任务合同

**执行模型**：

- 无依赖的任务并行启动
- 有依赖的任务等待前序完成后启动，**前序结果自动注入为后序任务的上下文**
- 同一 Worker 的多个任务串行执行（避免自身并发），不同 Worker 的任务并行执行（共享工作空间，见 8.1）

**同 Worker 排队策略**：

当多个任务路由到同一 Worker 时，按以下优先级排序：

1. **依赖拓扑优先**：被其他任务依赖的任务先执行（关键路径优化，尽早解除下游阻塞）
2. **同拓扑层级内按分配顺序（FIFO）**：无依赖关系的任务按 Phase A 的分配顺序执行

**Worker 执行中的上报机制**：

Worker 在执行中遇到以下情况时，应通过消息通道上报编排者：

| 上报场景 | Worker 动作 | 编排者响应 |
| -------- | ----------- | ---------- |
| 发现任务目标存在歧义 | 说明歧义点 + 给出自己的理解和倾向 | 澄清目标或确认 Worker 理解 |
| 发现与其他任务的潜在冲突 | 说明冲突内容 | 协调相关任务或调整契约 |
| 任务实际复杂度远超预期 | 说明原因 + 是否需要拆分 | 决定继续/拆分/升级 |
| 遇到不可恢复的技术阻塞 | 说明阻塞原因 + 已尝试的方案 | 决定改派/调整目标/上报用户 |

**不需要上报的情况**：

- 工作范围自然扩展（读了更多文件、修改了相邻模块）
- 技术方案选择（选用哪种算法、哪个库）
- 执行顺序调整（先测试后实现、先重构后修复）

### 7.3 Phase C：汇总与交付

**输入**：所有 Worker 的执行结果

**编排者动作**：

1. **收集结果**：获取每个 Worker 的产出（改动内容、执行摘要、异常记录）
2. **审计改动**：检查实际改动是否合理
3. **验收评估**：逐项检查每个任务的验收标准是否达成
4. **合成交付**：生成用户可理解的交付摘要
5. **处置未达标项**：对未通过验收的任务做恢复决策（见第 9 节）

**改动审计标准**：

| 审计维度 | 正常 | 需关注 | 需干预 |
| -------- | ---- | ------ | ------ |
| 改动范围 | 集中在任务相关模块 | 涉及相邻模块但有合理原因 | 大面积改动与任务目标无关的代码 |
| 改动性质 | 目标导向的增改 | 附带的小幅重构 | 未经授权的架构变更 |
| 跨任务影响 | 不影响其他任务的工作区域 | 修改了共享代码但不破坏契约 | 破坏了其他任务依赖的接口 |

审计结论：

- **正常**：直接纳入交付
- **需关注**：纳入交付，在摘要中标注供用户知晓
- **需干预**：暂不纳入交付，编排者决定回退/追加任务/上报用户

## 8. 多 Worker 协作

当一个用户需求需要多个 Worker 并行处理时，编排者承担协调职责。

### 8.1 共享工作空间

所有 Worker 共享同一个工作空间（同一 git 仓库工作目录），不做文件系统级隔离。

**设计理由**：软件工程任务中，多个任务常涉及共享代码（公共模块、配置文件、类型定义等）。隔离工作空间（如 git worktree/分支）虽然避免了并发冲突，但引入了合并复杂性，且同一份共享代码可能被不同 Worker 各自修改一遍，产生重复甚至矛盾的改动。共享工作空间让串行 Worker 天然继承前序改动，并行 Worker 对共享代码的修改冲突能在 Phase C 被集中发现和处理。

**共享工作空间下的冲突预防**：

编排者在 Phase A 拆分任务时，应主动进行**文件级分区**：

- **理想情况**：每个并行任务的 `scope_hint` 不重叠（前端任务聚焦前端文件，后端任务聚焦后端文件）
- **有重叠时**：通过 `depends_on` 将有文件交叉的任务串行化，让后序任务基于前序结果继续工作
- **不可避免的共享修改**：在协作契约中声明冻结区域，约束 Worker 不修改对方正在操作的共享文件

冲突预防的优先级：**分区 > 串行化 > 冻结声明**。能通过任务拆分避免的冲突，不应依赖运行时约束。

### 8.2 协作契约

协作契约定义 Worker 之间的交互约定，由编排者在 Phase A 制定：

**契约内容**：

- **接口定义**：如果 Worker A 的产出被 Worker B 消费，明确数据结构、函数签名、文件路径等
- **变更约束**：哪些共享代码/接口在本次编排中不可修改（冻结区域）
- **时序关系**：依赖方在被依赖方完成后才启动，编排者负责注入前序结果

### 8.3 契约冲突处理

当 Worker 的产出与契约不符时：

| 冲突类型 | 发现时机 | 处理方式 |
| -------- | -------- | -------- |
| 接口不匹配（参数/返回值与约定不同） | Phase C 审计 | 判定哪方更合理，要求偏差方修正 |
| 冻结区域被修改 | Phase C 审计 | 回退变更或重新协调契约 |
| 并行任务产生文件冲突 | Phase C 合并 | 编排者裁定保留哪个版本或合并 |

### 8.4 中间结果传递

Worker 之间不直接通信。所有信息流经编排者：

```text
Worker A 完成 → 结果返回编排者 → 编排者提取关键信息 → 注入 Worker B 的上下文
```

编排者在注入时应做**信息裁剪**：只传递 Worker B 需要的部分（接口定义、关键决策、产出路径），而非 Worker A 的完整执行日志。这控制了下游 Worker 的上下文规模，避免信息过载。

## 9. 失败处理

### 9.1 失败分类与恢复策略

| 失败类型 | 恢复策略 | 信息传递 |
| -------- | -------- | -------- |
| Worker 执行错误 | 分析错误 → 重试或改派 | 将错误信息 + 已完成部分传递给下一个执行者 |
| Worker 不可用 | 降级改派到备选 Worker | 原任务合同不变，附加"因 X 不可用而改派"的上下文 |
| 部分成功 | 接受已完成部分 + 创建补充任务 | 补充任务的上下文包含前序产出和剩余目标 |
| 契约冲突 | 编排者裁定 → 调整契约或重新执行 | 将冲突详情和裁定结果传递给需要修正的 Worker |
| 超时 | 收集已有产出 → 决定继续/终止 | 如继续，将已有产出和时间约束传递给续执行者 |

### 9.2 恢复执行原则

- **上下文传递**：恢复执行（重试/改派/补充）时，必须将前次执行的关键信息注入新任务上下文，包括：已完成的改动、失败原因、已排除的方案。目的是避免新执行者重复走入相同死胡同。
- **幂等性考量**：重试前评估前次执行是否产生了副作用（文件已修改、外部状态已变更）。如有副作用，在新任务上下文中明确说明当前状态，而非假设初始状态。
- **改派时的能力匹配**：降级改派时优先选择同类能力的 Worker，而非随机分配。如果没有同类能力可用，编排者应上报用户而非强行派给不匹配的 Worker。

### 9.3 失败升级

当编排者自身无法做出恢复决策时，必须升级到用户：

- 所有 Worker 均不可用
- 同一任务连续失败超过阈值
- 失败原因超出编排者理解能力（如外部服务故障、权限问题）
- 恢复操作需要用户授权（如回退已提交的代码）

升级时提供：失败任务摘要、已尝试的恢复动作、可选的下一步方案。

## 10. 用户交互模型

用户不仅是需求发起者，也是流程中的关键决策者。编排者需要在"自主执行"和"征询用户"之间取得平衡。

### 10.1 决策权分配

| 决策类型 | 编排者自主 | 需征询用户 |
| -------- | ---------- | ---------- |
| 任务拆分方式 | 是（用户可在交付后反馈） | 当需求严重模糊时 |
| Worker 路由选择 | 是（基于分类自动路由） | — |
| 任务改派（降级） | 是（自动降级 + 通知用户） | — |
| 需求歧义澄清 | — | 是（无法合理推断时） |
| 超出预期的大范围改动 | — | 是（在执行前确认） |
| 不可逆外部操作 | — | 是（如发布、数据库变更） |
| 失败恢复策略 | 首次自动恢复 | 连续失败时升级 |

### 10.2 用户可见性

用户在整个流程中应能获得以下信息：

**Phase A**：

- 编排者如何理解了用户需求
- 拆分了几个任务，分别交给哪个 Worker
- 任务间的依赖关系

**Phase B**：

- 各 Worker 的执行状态（进行中 / 已完成 / 失败）
- Worker 的阶段性产出或进度摘要
- 异常事件通知（改派、上报）

**Phase C**：

- 交付摘要：做了什么、改了什么、验收结果
- 异常标注：哪些地方有改动审计关注项
- 后续建议：如果有遗留问题，建议的下一步

### 10.3 用户中途干预

用户可以在流程的任何阶段发起干预：

| 干预类型 | 编排者处理 |
| -------- | ---------- |
| 修改需求 | 评估变更影响 → 对进行中的任务发送补充指令或终止重派 |
| 取消任务 | 终止相关 Worker → 收集已有产出 → 向用户报告当前状态 |
| 追加约束 | 向进行中的 Worker 发送补充约束（如果 Worker 支持运行时指令接收） |
| 优先级调整 | 调整待执行任务的排序 |

## 11. 质量度量

度量以**结果**为导向：

| 指标 | 含义 | 目标方向 |
| ---- | ---- | -------- |
| 一次交付成功率 | 任务无需返工即通过验收的比例 | 上升 |
| 协作返工率 | 因契约不清导致的重新执行比例 | 下降 |
| 改派率 | Worker 不可用触发降级改派的比例 | 下降 |
| 端到端耗时 | 从用户发起到交付完成的时间 | 下降 |
| Worker 利用率 | 各 Worker 的实际执行占比 | 均衡 |
| 用户干预率 | 用户在执行过程中主动干预的比例 | 适中 |
| 失败恢复成功率 | 自动恢复（无需用户介入）解决失败的比例 | 上升 |

**关于"用户干预率"**：这个指标的目标不是最低化，而是合理化。干预率过高说明编排者的自主决策能力不足或需求理解不准确；干预率过低可能意味着用户对过程缺乏感知，或编排者在应该征询时没有征询。

## 12. 反模式

| 反模式 | 危害 | 正确做法 |
| ------ | ---- | -------- |
| 编排者写施工图，Worker 只执行步骤 | Worker 智力浪费，结果质量取决于编排者的实现能力而非 Worker 的专业能力 | 编排者定义目标和验收，Worker 自主决定实现 |
| Worker 被边界卡死，交付不完整结果 | 用户收到不可用的半成品 | 可逆操作默认放行，事后审计异常扩散 |
| Worker 悄悄越界，编排者不知情 | 改动范围不可控，可能引入意外破坏 | Phase C 审计改动范围，异常时追溯 |
| 所有任务同等对待 | 简单任务浪费编排开销，复杂任务缺少必要的契约和协调 | 按 L1/L2/L3 分级，匹配合同详细度 |
| 编排者在 Worker 执行中频繁干预 | 破坏 Worker 自主性，增加延迟 | 信任 Worker 自主决策，Phase C 统一评估 |
| 失败后无信息传递，重试盲目重来 | 重复踩坑，浪费资源 | 保留失败上下文，注入到恢复任务中 |
| 编排者从不征询用户 | 需求偏差在 Phase C 才暴露，全部返工 | 对高歧义需求在 Phase A 主动征询 |
| 编排者什么都问用户 | 用户体验差，编排者缺乏自主价值 | 按 10.1 决策权分配表区分自主与征询 |

## 13. 核心原则

1. **编排者**定义"做什么 + 怎么验收 + 协作约束"
2. **Worker** 决定"怎么做 + 为什么这么做 + 结果证明"
3. **硬边界**靠系统强制，**可逆操作**靠事后审计，**不可逆操作**靠工具层拦截，**协作边界**靠编排者协调
4. **少拆优于错拆**，不确定的复杂性应留给 Worker 处理
5. **不确定时向下兼容**，宁可多给上下文也不给模糊合同
6. **失败是常态**，恢复策略比预防策略更重要，上下文传递比盲目重试更重要
7. **用户是决策者不是旁观者**，编排者在自主与征询之间保持平衡
