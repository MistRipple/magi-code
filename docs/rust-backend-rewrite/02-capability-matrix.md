# Rust 后端重构能力对照表

更新时间：2026-04-17

> 本文档用于维护“当前 Magi 后端能力清单”与“Rust 重构覆盖度”的对照关系。
>
> 任何能力域开始重构前，都应先在本表中登记；任何能力域准备切换前，都必须以本表为依据确认覆盖度。

---

## 1. 状态定义

- `未开始`：尚未进入 Rust 重构范围
- `建模中`：正在定义领域模型、协议、边界
- `开发中`：已有 Rust 实现，但能力未闭环
- `已覆盖`：Rust 实现已覆盖核心能力
- `待验证`：已覆盖，但尚未完成切换前验证
- `已达切换条件`：具备切换评估资格

---

## 2. 真相源级别说明

- `入口协调层`：对外暴露入口，但不应长期承担核心状态真相
- `核心真相源`：后端必须长期持有、持久化或恢复的业务真相
- `核心治理层`：策略、审批、风险与安全边界的统一真相
- `核心执行层`：工具、执行与隔离语义的统一真相
- `核心审计层`：事件、审计、用量、回放的统一真相
- `派生读模型`：由核心真相源派生出的查询视图或投影视图
- `边界桥接层`：与外部模型、MCP、宿主交互的边界层
- `扩展执行层`：在核心执行层基础上附着的扩展机制

## 4. 使用规则

1. 每个能力域状态变化时，必须同步更新本表
2. 任一能力域若发现旧实现存在严重语义错误，应同时登记到“语义偏差台账”
3. 若某能力域在 Rust 侧已实现，但无法完成“切换前必须验证”中的关键项，不得标记为“已达切换条件”
4. 若能力域横跨多个旧模块，必须先在本表中确认新的主归属，再开始写 Rust 实现

## 6. 当前结论

当前已经完成 `P0` 首批骨架建模，并且 `magi-daemon`、`magi-api`、`magi-session-store`、`magi-workspace`、`magi-governance`、`magi-event-bus` 均已进入“开发中”状态。

但从现有代码结构看，真正决定 Rust 后端能否成立的，不是 UI，也不是宿主壳，而是以下几条主链是否能先被稳定收口：

1. `Agent API / Session / Workspace / Snapshot`
2. `Mission / Dispatch / Worker / Tool`
3. `Governance / Event / Audit / Usage`
4. `Knowledge / Memory / Context`

后续任何能力域若脱离这四条主链单独重构，风险都会显著升高。
