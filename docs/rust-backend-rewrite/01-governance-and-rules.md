# Rust 后端重构治理与执行规则

更新时间：2026-04-15

---

## 1. 当前重构模式

当前采用的不是“渐进接管现有运行链路”，而是：

> 本地影子重构 Rust 后端，完成后统一评估切换。

当前阶段明确约束：

1. Rust 后端先不接现有前端
2. Rust 后端先不接现有宿主壳
3. Rust 后端先不接现有运行时真相源
4. 现有系统继续作为唯一运行版本
5. Rust 重构版只作为本地重构工程推进

---

## 2. 重构边界

当前重构边界只覆盖后端相关能力域：

- Agent API
- Session / Workspace / Snapshot / Recovery
- Orchestrator / Task / Todo / Worker Runtime
- Builtin Tools / Tool Policy / Sandbox / Approval
- MCP / Skill 边界设计
- Knowledge / Memory / Context
- Governance / Audit / Usage / Event Bus

当前不在重构边界内的内容：

- 现有 UI 展示层改造
- 宿主壳接线
- 现网切换
- 运行期双后端兼容

---

## 3. 根因级重构原则

本次 Rust 重构不是 TypeScript 代码搬运工程。

如果在重构过程中发现当前实现存在以下问题，必须允许并推动结构性重构：

- 职责混杂
- 架构边界错误
- 单模块过大
- 状态重复表达
- 双轨语义
- 无效兼容分支
- 为历史包袱堆出来的补丁式逻辑

必须遵循：

1. 以目标结构为准，不以旧代码形态为准
2. 以根本修正为准，不以补丁搬运为准
3. 能拆分则拆分，能删除则删除，能收敛则收敛
4. 禁止把明显不合理的旧实现原样带入新后端

---

## 4. 协议与语义优先级

当发现旧实现不合理时，判定优先级如下：

1. 目标领域模型
2. 稳定对外契约
3. 旧代码实现细节

说明：

- 内部结构应以新后端目标语义为主
- 对外稳定契约需要单独评估是否兼容
- 错误的旧行为、混乱状态、历史补丁不构成迁移标准

---

## 5. 必须遵循 `cn-engineering-standard`

后续本目录对应的所有重构任务，都必须遵循
[$cn-engineering-standard](/Users/xie/.codex/skills/cn-engineering-standard/SKILL.md)。

最低执行要求：

- 全程中文沟通
- 根因导向
- 严格限制改动范围
- 禁止多重实现
- 禁止回退逻辑
- 禁止补丁式修复
- 完成后必须清理废弃代码
- 执行“发现-修复-清理-测试-验证”闭环

这不是建议项，而是强制约束。

---

## 6. 本地影子重构的风险控制

本地影子重构最大风险不是“现在接不了现网”，而是：

1. 范围失控
2. 能力遗漏
3. 语义偏离
4. 最终无法切换

因此必须长期维护：

- 能力对照表
- 语义偏差台账
- crate 映射表
- 阶段出口门槛

这四类文档缺一不可。

---

## 7. 当前禁止事项

1. 禁止在 Rust 重构版里偷偷接现有前后端链路
2. 禁止为了“先跑通”保留回退逻辑
3. 禁止形成新旧双真相源
4. 禁止先写实现、后补语义定义
5. 禁止脱离能力对照表单独发散式重写

---

## 8. 当前执行结论

当前后端重构的正确策略是：

> 研发上并行，运行上隔离，切换上统一。

这条原则在本目录全部文档中默认成立。
