# MultiCLI 文件快照系统架构分析 v2 (准确版)

## 设计意图理解

### 核心设计

**快照策略**: 单一检查点模式
```
任务前: 创建快照保存初始状态
  ↓
并行任务执行 (可能同时修改同一文件)
  ↓
FileLockManager 保证写入顺序
  ↓
用户自主选择何时确认变更
  ↓
确认后: 删除旧快照 + 立即创建新快照 (当前状态成为新基准)
```

**并发控制**: 双重保护
```
1. TaskDependencyGraph (静态): 调度前检测文件冲突,串行化调度
2. FileLockManager (动态): 运行时加锁,保证写入顺序

为什么两者都需要?
- TaskDependencyGraph 可能遗漏动态生成的文件变更
- FileLockManager 提供运行时保护,防御性编程
```

这个设计是**合理的**。

## 实际问题分析

### 🔴 P0 问题: 快照创建时机错误

**当前代码**:
```typescript
// worker-agent.ts:284-292
const response = await this.executeCLI(prompt);  // ← 执行任务

// 在任务执行后创建快照!
if (response.fileChanges && response.fileChanges.length > 0 && this.snapshotManager) {
  for (const change of response.fileChanges) {
    try {
      this.snapshotManager.createSnapshot(change.filePath, this.type, subTask.id);
    } catch (err) {
      console.warn(`[WorkerAgent] 创建快照失败: ${change.filePath}`, err);
    }
  }
}
```

**问题**:
1. **时机错误**: 在 `executeCLI()` **之后**创建快照
2. **保存的不是初始状态**: 此时文件已被修改,快照保存的是**修改后的内容**!
3. **回滚失效**: 快照内容 = 当前内容,回滚后文件不变

**后果示例**:
```
初始状态: file.ts 内容为 "A"
  ↓
Task 执行: file.ts 修改为 "B"
  ↓
创建快照: 保存内容 "B" ← 错误!应该保存 "A"
  ↓
Task 失败回滚: 恢复快照 "B" → file.ts 仍然是 "B"
  ↓
结果: 回滚失败,文件未恢复到初始状态!
```

**根因**: 快照创建逻辑放在了 `executeCLI()` 之后

### 🔴 P0 问题: 确认后未创建新快照

**当前代码**:
```typescript
// snapshot-manager.ts:284-310
acceptChange(filePath: string): boolean {
  // 1. 删除快照文件
  fs.unlinkSync(snapshotFile);

  // 2. 从 session 中移除快照
  this.sessionManager.removeSnapshot(session.id, relativePath);

  // 3. 缺失: 为当前状态创建新快照!

  return true;
}
```

**问题**: 确认后只删除旧快照,未创建新基准快照

**后果示例**:
```
初始状态: file.ts = "A", 快照保存 "A"
  ↓
Task1 执行: file.ts = "B"
  ↓
用户确认变更: 删除快照 "A" ← 但未创建快照 "B"!
  ↓
Task2 执行: file.ts = "C"
  ↓
Task2 失败回滚: 无快照可回滚! (快照已被删除)
  ↓
结果: 用户想回到 "B",但只能手动撤销或丢失
```

**根因**: `acceptChange()` 未实现"创建新基准快照"逻辑

### 🟡 P1 问题: 快照依赖 LLM 声明,无法捕获实际变更

**当前代码**:
```typescript
// worker-agent.ts:284-292
if (response.fileChanges && response.fileChanges.length > 0) {
  for (const change of response.fileChanges) {
    this.snapshotManager.createSnapshot(change.filePath, this.type, subTask.id);
  }
}
```

**问题**: 依赖 LLM 返回的 `fileChanges`,而非文件系统实际监控

**风险**:
- LLM 可能遗漏某些文件 (如间接依赖)
- LLM 可能声明错误的文件路径
- 无法检测 LLM 未声明但实际修改的文件

**后果**: 部分文件修改无快照,回滚不完整

---

### 🟡 P1 问题: FileLockManager 与 TaskDependencyGraph 协调不足

**潜在问题**: 两个机制独立工作,可能出现:

1. **死锁风险**:
```
DAG 调度:
  Batch 1: Task A (file1) + Task B (file2)
  Batch 2: Task C (file1, file2)

FileLockManager:
  Task A 获取 file1 锁
  Task B 获取 file2 锁
  Task C 等待 file1 和 file2 → 可能永远等待 (如果 A/B 不释放)
```

2. **优先级冲突**:
```
DAG: 基于依赖关系排序
FileLockManager: 基于优先级 + 饥饿避免排序

两者可能冲突,导致执行顺序不符合预期
```

**当前代码是否有问题?** 需要验证:
- DAG 执行是否等待锁释放?
- 锁超时机制是否存在?

### 修复 2: 确认后创建新基准快照 (P0)

**目标**: 用户确认后,当前状态成为新基准,创建新快照

**修改**: `src/snapshot-manager.ts`

```typescript
/** 接受变更（删除旧快照，创建新基准快照） */
acceptChange(filePath: string): boolean {
  const session = this.sessionManager.getCurrentSession();
  if (!session) return false;

  const relativePath = path.isAbsolute(filePath)
    ? path.relative(this.workspaceRoot, filePath)
    : filePath;

  const snapshot = this.sessionManager.getSnapshot(session.id, relativePath);
  if (!snapshot) return false;

  // 1. 删除旧快照文件
  const oldSnapshotFile = path.join(this.getSnapshotDir(session.id), `${snapshot.id}.snapshot`);
  if (fs.existsSync(oldSnapshotFile)) {
    fs.unlinkSync(oldSnapshotFile);
  }

  // 2. 从 session 中移除旧快照元数据
  this.sessionManager.removeSnapshot(session.id, relativePath);

  // ========================================
  // ✅ 新增: 创建新基准快照
  // ========================================
  const absolutePath = path.join(this.workspaceRoot, relativePath);

  // 安全检查
  const normalizedPath = path.normalize(absolutePath);
  const normalizedRoot = path.normalize(this.workspaceRoot);
  if (!normalizedPath.startsWith(normalizedRoot)) {
    console.error(`[SnapshotManager] Path traversal detected: ${filePath}`);
    return false;
  }

  // 读取当前文件内容 (确认后的状态)
  let currentContent = '';
  if (fs.existsSync(absolutePath)) {
    currentContent = fs.readFileSync(absolutePath, 'utf-8');
  }

  // 创建新快照 ID
  const newSnapshotId = generateId();
  const newSnapshotMeta: FileSnapshotMeta = {
    id: newSnapshotId,
    filePath: relativePath,
    lastModifiedBy: snapshot.lastModifiedBy,
    lastModifiedAt: Date.now(),
    subTaskId: snapshot.subTaskId, // 继承原 subTaskId
  };

  // 保存新快照内容
  const newSnapshotFile = path.join(this.getSnapshotDir(session.id), `${newSnapshotId}.snapshot`);
  fs.writeFileSync(newSnapshotFile, currentContent, 'utf-8');

  // 添加新快照元数据
  this.sessionManager.addSnapshot(session.id, newSnapshotMeta);

  console.log(`[SnapshotManager] 确认变更并创建新基准快照: ${relativePath}`);

  globalEventBus.emitEvent('snapshot:accepted', {
    sessionId: session.id,
    data: {
      filePath: relativePath,
      oldSnapshotId: snapshot.id,
      newSnapshotId: newSnapshotId,
    },
  });

  return true;
}
```

**验证**:
- ✅ 旧快照被删除
- ✅ 新快照保存确认后的状态
- ✅ 后续任务可以回滚到新基准

### 修复 3: 添加文件监控,验证 LLM 声明 (P1)

**目标**: 检测 LLM 未声明但实际修改的文件

**新增**: `src/utils/file-watcher.ts`

```typescript
import * as chokidar from 'chokidar';

export class FileWatcher {
  private watcher: chokidar.FSWatcher | null = null;
  private changedFiles = new Set<string>();
  private watching = false;

  /**
   * 开始监控指定文件
   */
  start(files: string[]): void {
    if (this.watching) {
      throw new Error('FileWatcher already started');
    }

    this.watcher = chokidar.watch(files, {
      persistent: false,
      ignoreInitial: true,
      awaitWriteFinish: {
        stabilityThreshold: 100,
        pollInterval: 50,
      },
    });

    this.watcher.on('change', (path) => {
      this.changedFiles.add(path);
      console.log(`[FileWatcher] 检测到文件变更: ${path}`);
    });

    this.watcher.on('add', (path) => {
      this.changedFiles.add(path);
      console.log(`[FileWatcher] 检测到新文件: ${path}`);
    });

    this.watching = true;
  }

  /**
   * 获取实际变更的文件列表
   */
  getChangedFiles(): string[] {
    return Array.from(this.changedFiles);
  }

  /**
   * 停止监控
   */
  async stop(): Promise<void> {
    if (this.watcher) {
      await this.watcher.close();
      this.watcher = null;
    }
    this.watching = false;
  }

  /**
   * 重置变更记录
   */
  reset(): void {
    this.changedFiles.clear();
  }
}
```

**集成到 WorkerAgent**:

```typescript
// worker-agent.ts
import { FileWatcher } from '../utils/file-watcher';

async executeSubTask(subTask: SubTask, taskId: string): Promise<ExecutionResult> {
  const startTime = Date.now();

  try {
    // 1. 创建快照
    if (subTask.targetFiles && subTask.targetFiles.length > 0 && this.snapshotManager) {
      for (const file of subTask.targetFiles) {
        this.snapshotManager.createSnapshot(file, this.type, subTask.id);
      }
    }

    // 2. 启动文件监控
    const watcher = new FileWatcher();
    if (subTask.targetFiles && subTask.targetFiles.length > 0) {
      watcher.start(subTask.targetFiles.map(f =>
        path.isAbsolute(f) ? f : path.join(this.workspaceRoot, f)
      ));
    }

    // 3. 执行任务
    const response = await this.executeCLI(prompt);

    // 4. 停止监控并验证
    await watcher.stop();
    const actualChanges = watcher.getChangedFiles();
    const declaredChanges = (response.fileChanges || []).map(c => c.filePath);

    // 5. 检测差异
    const undeclared = actualChanges.filter(file =>
      !declaredChanges.some(declared => file.endsWith(declared))
    );

    if (undeclared.length > 0) {
      console.warn(
        `[WorkerAgent ${this.id}] LLM 未声明的文件变更:\n` +
        undeclared.map(f => `  - ${f}`).join('\n')
      );

      // 为未声明的文件补充快照 (如果还没有)
      for (const file of undeclared) {
        try {
          this.snapshotManager?.createSnapshot(file, this.type, subTask.id);
        } catch (err) {
          // 可能已经有快照了,忽略错误
        }
      }
    }

    // 6. 返回结果
    const result: ExecutionResult = {
      workerId: this.id,
      workerType: this.type,
      taskId,
      subTaskId: subTask.id,
      result: this.formatResultContent(subTask, response.content || response.raw),
      success: !response.error,
      duration: Date.now() - startTime,
      modifiedFiles: [...new Set([...declaredChanges, ...undeclared])], // 合并声明和实际
    };

    return result;
  } catch (error) {
    // ... 错误处理
  }
}
```

### 修复 4: 增强 FileLockManager 超时和死锁检测 (P1)

**目标**: 防止锁等待导致的死锁或无限等待

**修改**: `src/orchestrator/file-lock-manager.ts`

```typescript
export class FileLockManager extends EventEmitter {
  private locks: Map<string, string> = new Map();
  private queue: LockRequest[] = [];
  private requestCounter = 0;

  // ✅ 新增: 死锁检测配置
  private readonly MAX_WAIT_TIME = 30000; // 30秒超时
  private readonly DEADLOCK_CHECK_INTERVAL = 5000; // 5秒检测一次

  constructor() {
    super();
    // 启动死锁检测定时器
    setInterval(() => this.checkDeadlock(), this.DEADLOCK_CHECK_INTERVAL);
  }

  async acquire(
    files: string[],
    priority?: number,
    abortSignal?: AbortSignal
  ): Promise<() => void> {
    const normalized = this.normalizeFiles(files);

    if (normalized.length === 0) {
      return () => undefined;
    }

    if (abortSignal?.aborted) {
      throw new Error('任务已取消');
    }

    return new Promise((resolve, reject) => {
      const request: LockRequest = {
        id: String(++this.requestCounter),
        files: normalized,
        priority: priority ?? DEFAULT_PRIORITY,
        enqueuedAt: Date.now(),
        resolve,
        reject,
        abortSignal,
      };

      // ✅ 新增: 超时检测
      const timeoutId = setTimeout(() => {
        this.removeRequest(request.id);
        reject(new Error(
          `文件锁获取超时 (${this.MAX_WAIT_TIME}ms): ${normalized.join(', ')}`
        ));
      }, this.MAX_WAIT_TIME);

      // 成功获取锁后清除超时
      const originalResolve = request.resolve;
      request.resolve = (release) => {
        clearTimeout(timeoutId);
        originalResolve(release);
      };

      if (this.tryAcquire(request)) {
        resolve(() => this.release(request));
        return;
      }

      if (abortSignal) {
        const onAbort = () => {
          clearTimeout(timeoutId);
          this.removeRequest(request.id);
          const reason = abortSignal.reason;
          reject(reason instanceof Error ? reason : new Error('任务已取消'));
        };
        request.abortHandler = onAbort;
        abortSignal.addEventListener('abort', onAbort, { once: true });
      }

      this.queue.push(request);
    });
  }

  // ✅ 新增: 死锁检测
  private checkDeadlock(): void {
    const now = Date.now();

    for (const request of this.queue) {
      const waitTime = now - request.enqueuedAt;

      if (waitTime > this.MAX_WAIT_TIME * 0.8) {
        console.warn(
          `[FileLockManager] 检测到长时间等待:\n` +
          `  请求 ID: ${request.id}\n` +
          `  等待时间: ${waitTime}ms\n` +
          `  等待文件: ${request.files.join(', ')}\n` +
          `  当前锁状态: ${this.debugLockState()}`
        );
      }
    }
  }

  // ✅ 新增: 调试锁状态
  private debugLockState(): string {
    const locks = Array.from(this.locks.entries())
      .map(([file, owner]) => `${file} → ${owner}`)
      .join(', ');
    return locks || '(无锁)';
  }

  // ... 其他方法保持不变
}
```

## 架构评估 (修正后)

### 设计合理性

**优点** ✅:
1. **双重保护机制**: TaskDependencyGraph + FileLockManager,防御性强
2. **用户自主控制**: 快照确认时机由用户决定,灵活性高
3. **单一检查点**: 快照不是版本链,而是回滚检查点,设计清晰

**缺点** ❌:
1. **快照时机错误**: 事后创建快照,保存的不是初始状态
2. **确认后缺失新快照**: 用户确认后未创建新基准
3. **依赖 LLM 声明**: 无文件监控,可能遗漏实际变更
4. **无死锁防护**: FileLockManager 缺少超时和死锁检测
