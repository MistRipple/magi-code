## 1. 角色定位：从“自由对话”到“工程调度”

传统 AI 助手往往是单体执行者，而 Magi 是一个分工明确的**微服务化执行矩阵**。必须认清两种核心角色的边界：

### 1.1 编排者（Orchestrator）：目标管理者与任务派发器
编排者负责全局调度，将模糊的用户意图转化为可验收的明确任务。

- **必须做**：结合本地项目知识（PKB）理解用户意图；按业务目标拆分并派发子任务（Mission）；提供代码“寻路点”作为初始 Context；跟踪并验收任务结果。
- **不应做**：指定具体代码行级别的实现步骤；微操 Worker 的思考过程；直接干预底层文件的并发修改。
- **任务拆分原则**：按**“要达成什么结果”**来拆分，而不是按“先做 A 再做 B”来拆分。宁可给大块任务，让 Worker 内部自行探索。

### 1.2 执行者（Worker Personas）：自主工程沙盒
Worker 是具体的干活角色，对其承包的任务拥有完全的技术决策权。

- **自主探索**：负责基于收到的上下文，通过 `codebase_retrieval` 和 `grep_search` 探索代码细节。
- **零冲突执行**：自主选择实现方案，申请底层的文件锁（File Mutex）并执行意图驱动的代码修改。
- **自纠错 (Self-Correction)**：必须进行自我验证（如编译、查错），通过动态追加 Todo 实现自我修复，无需事事向编排者汇报失败。并在结束时向编排者提交包含总结和异常的交付报告。

## 2. 上下文与任务流转 (Context & Mission Flow)

### 2.1 任务合同 (Mission Contract) 流转
所有的工作都通过“任务合同”的形式在 Orchestrator 和 Worker 之间传递。一个标准且解耦的任务包含：
1. **Goal（目标）**：明确的技术指标或业务结果。
2. **Acceptance（验收）**：判断任务是否完成的刚性标准。
3. **Context（上下文）**：由编排者收集好的文件线索、前置条件，精准注入，防止新拉起的 Worker 盲人摸象。

### 2.2 纯本地防御性上下文体系
Magi 抛弃了臃肿且容易导致幽灵记忆的外部向量库，打造了三维立体的本地上下文防御体系：
- **项目知识基建 (PKB)**：项目级的架构约定 (ADR) 和自动沉淀的经验库。
- **寻路点注入**：编排者通过精准检索定位“寻路点”，而非将整个工作区的代码全部塞给 Worker。
- **滚动摘要提纯 (Rolling Summary)**：系统在长任务中通过 `MemoryDocument` 机制压缩会话上下文，排除死胡同日志，控制 Token 消耗并保持重点不丢失。

## 3. 动态双模式治理 (Adaptive Mode Governance)

系统根据用户选择的模式，动态调整编排者的代码权限、验证轮次以及执行预算，在**速度**与**安全性**之间取得完美平衡：

### 3.1 常规模式（Standard Mode）
- **特点与适用场景**：快节奏，适合轻量修改、常规代码调整和日常问答。响应时间在秒级到数十秒。
- **执行策略（软约束）**：编排者拥有**直接修改文件**的能力（限制最多修改 3 个文件）。对于简单任务，编排者可直接介入而无需强制派发任务，减少拆解和上下文传递的损耗。
- **验收机制**：轻量级，通常经过 1~2 轮的意图验证即可快速返回结果。

### 3.2 深度模式（Deep Mode）
- **特点与适用场景**：极高安全性、长耗时，专为系统级重构、跨模块特性开发和深水区 Bug 排查设计。
- **执行策略（硬约束）**：编排者**被严格禁止直接修改代码**。所有的工程读写操作必须通过创建 TaskView，并把具体的子 Todo 委派给专门的 Worker 角色来执行。
- **验收机制**：重度闭环。Worker 拥有远高于常规模式的工具调用预算（最高达 8 轮交互），允许其通过反复编译、查错、利用快照回滚（Snapshot Recovery）并重写，保障复杂重构的完整性与安全性。

## 4. 安全与并发控制 (Safety & Concurrency Control)

当多个 Worker 同时运行或任务交错时，Magi 底层依靠硬核基建保障工程安全：

### 4.1 零冲突底层（Zero-Collision Engine）
- **意图驱动编辑**：抛弃极易失效的“行号匹配”，大模型仅输出“修改意图”，由底层解析器精准定位重写。
- **实时强读**：写入动作发生瞬间，底层系统强制读取磁盘最新状态，彻底避免由于 Git 延迟或并发修改导致的上下文行号偏移。
- **文件级互斥锁 (File-level Mutex Lock)**：这是防冲撞的核心。基于物理路径的排他锁，确保同一时刻只有一个 Agent 能写入特定文件，从根本上消除了多模型并发覆盖代码的毁灭性灾难。

### 4.2 物理状态隔离
所有的执行日志、通信变量和上下文更新，都在底层的 `UnifiedSessionManager` 中与具体的 `traceId` 或 `sessionId` 强绑定。防止跨任务并发时产生消息串接和幽灵污染。

## 5. 协作闭环的生命周期 (Collaboration Lifecycle)

典型的 Magi 任务运转生命周期完美复刻了专业工程团队的协作流：

1. **Phase A：规划与派发（Orchestrator）**
   - 读取用户输入和 PKB 经验库，通过本地检索快速收集线索。
   - 根据两模式治理决定：自己直接动手（常规）还是拆分并发拓扑图（深度）。
   - 派发时，注入整理好的上下文寻路点。

2. **Phase B：自主执行与纠错（Worker）**
   - 接收任务，进入沙盒，在局部代码展开深度探索。
   - 自主思考和执行逻辑修改（全程受到 File Lock 和 Snapshot 快照备份的保护）。
   - **闭环内纠错**：如果不幸编译报错，Worker 会利用高昂的执行预算，在本地动态生成子 Todo 挂起并继续自我修复，绝不轻易向 Orchestrator 汇报“我失败了”。
   - 最终整理修改总结与剩余异常点上报。

3. **Phase C：验收与闭环（Orchestrator）**
   - 汇总并分析所有并发 Worker 的交付报告。
   - 检查任务是否达标。未达标则触发补充指令进行二次调度；达标则触发 `WisdomExtractor` 沉淀经验，归档滚动摘要并给用户呈现最终清爽的结论。

## 6. 核心反模式警告 (Anti-Patterns)

为了让这套精密的架构发挥最大效力，系统严厉禁止以下操作：

- ❌ **过度指挥 (Micromanagement)**：编排者给出极其具体的代码修改指令，剥夺 Worker 的探索权，让高智商模型沦为打字机。
- ❌ **上下文轰炸 (Context Bombing)**：把几万 Token 的历史试错日志未加提纯直接塞给新 Worker，导致 Worker 注意力失焦。
- ❌ **越权修改 (Privilege Escalation)**：在深度模式下，编排者试图绕过 Worker 直接自己通过 Edit 工具改代码（会被系统软熔断警告）。
- ❌ **失忆重试 (Amnesia Retry)**：任务失败后，不提取失败根因并注入到下一次的 Prompt 中，直接让 Worker 盲目重试陷入死循环。

---

> **结语**：
> 1. **编排者指路，Worker 铺轨**：编排者提供高层视野和线索，Worker 负责深挖细节。
> 2. **所见即所得**：抛弃滞后的外部云端缓存，完全依赖本地工作区的实时状态和本地搜索引擎。
> 3. **安全底线**：深度模式的强制委派机制，加上意图编辑与文件级排他锁，共同为复杂并发场景构筑了坚不可摧的代码安全屏障。
