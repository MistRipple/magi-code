# [开源] Magi — 把 VSCode 变成你的 AI 研发包工队（多智能体编排）

各位大佬好，给大家分享一个我最近在卷的开源项目 —— **Magi**。
它是一个直接跑在 VSCode 里的多智能体（Multi-Agent）编排系统，目前已经迭代到了 **v0.2.0** 版本。

项目地址：https://github.com/MistRipple/magi
文档：https://github.com/MistRipple/magi-docs
（支持 VSCode 1.93.0+）

## 🤔 痛点：为什么要做这个？

大家平时肯定重度依赖 AI 编程，Copilot、Cursor、Cline 等工具我都深度用过，但写着写着总觉得憋屈：

1. **“单体智能”的上下文硬伤**：聊久了，模型就开始丢三落四。一开始定好的架构规范，几轮代码改下来全忘了，甚至把原本跑通的功能给改挂。
2. **术业有专攻，但模型被绑定**：有的模型（比如 Claude 3.5 Sonnet）逻辑强、适合搞架构设计和深挖 Bug；有的模型（比如 Gemini 2.0 Pro）干前端切图、写文案很溜。但现有的 Cursor 往往一杆子捅到底，不能让最合适的模型做最合适的事。
3. **线性排队，效率瓶颈**：写业务逻辑 -> 补充单元测试 -> 调整 UI，这些明明可以并行的事，硬是被逼着一步步串行等。
4. **一键“改炸”的恐惧**：有时候让 AI 改个大需求，它咔咔顺手改了十几个文件，结果跑不起来。想退回去？手动挑文件还原简直是灾难。

**一句话总结：现有的 AI 编程助手像是一个需要你反复调教的实习生，而我们需要的是一个有分工的 AI 研发团队。**

## 💡 Magi 的解法：拆、分、合

Magi 的核心设计思路是把单体的 Chat 变成工程化的多节点协同：**意图洞察 -> 深度拆解 -> 异构协作**。

### 1. 三层自适应（不无脑上 Agent）

不是啥任务都要搞“多智能体”。Magi 的 Orchestrator 会先做复杂度嗅探：
- **L1 (即时响应)**：简单的问答和代码解释，秒级返回。
- **L2 (单步工具)**：需要读文件、跑测试的，单步执行。
- **L3 (全链路协作)**：遇到大需求，才启动任务拆解，分发给下级 Worker 们并行处理。

### 2. 异构 Worker 矩阵（模型混搭）

这是我最喜欢的一个 feature。Magi 提供 3 个 Worker 槽位，你可以**根据各家模型的长板和 API 价格**来自由组合“梦之队”：
- **大脑/规划师 (Orchestrator)**：接 Claude 4.6 Sonnet，负责读懂需求、拆分任务、最终验收。
- **干活/前端 (Worker 1)**：接 Gemini 3.0 Pro，负责写 UI 组件、补全琐碎逻辑（便宜量大）。
- **排错/修虫 (Worker 2)**：接 GPT-5.3-Codex 或其他小模型，专门负责跑测试和修 Bug。

### 3. 企业级协作流 & 安全网

- **自动契约 (Contracts)**：Worker 之间会自动约定接口字段和格式，不会出现 A 写了 `user_id` 而 B 调了 `userId` 的坑。
- **任务级快照 (Snapshot)**：这解决了“改炸”的痛点。AI 在动文件前会自动留底，不管改崩了多少个文件，点一下一键回溯到当前任务前的状态，免去复杂的 Git 操作。
- **共享上下文**：跨 Worker 变量与变更自动同步，从源头解决“信息孤岛”。早期版本中的上下文同步损耗和工具循环调用（套娃）问题在最新的 v0.2.0 中已经得到了妥善解决。

## 🛠 技术栈 & 现状

底层是用 TypeScript 和 VSCode Extension API 手撸的，界面基于 Svelte + TailwindCSS，协议层完整支持 **MCP (Model Context Protocol)** 以方便后续扩充工具链。
目前自带 15+ 基础能力（文件读写、Git、执行脚本等）。

目前版本是 v0.2.0，核心链路已经跑通并相对稳定。
